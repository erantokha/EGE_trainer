# REPORT — автономная разработка iOS-приложения EGE Trainer

Дата: 2026-06-11 (ночной автономный прогон).
Статус: **частично готово — цель B достигнута, бо́льшая часть цели A выполнена**;
единственный системный блокер — на машине сборки нет Xcode (детали в §4.1).

## 1. Что сделано

### 1.1. Исследование веб-продукта

- Локальный сайт поднят (`python3 -m http.server 8000`), карта экранов и RPC
  обеих ролей составлена по коду (`tasks/*.js`, `app/providers/*`) и реестру
  `docs/supabase/runtime_rpc_registry.md`.
- **Live-контракты сняты с продакшена**: логин обоих тестовых аккаунтов и все
  ключевые RPC вызваны через REST напрямую; Swift-модели написаны по реальным
  JSON-ответам, а не по предположениям.
- **19 мобильных web-reference скриншотов** (390×844, dpr2, мобильный UA) обеих
  ролей: `Screenshots/web-reference/` (скрипт: `reports/_ios_web_reference_shots.cjs`).
- Разобран механизм контента: ДЗ хранит только ссылки `{topic_id, question_id}`;
  текст/рисунок/ответ резолвятся из статических JSON `content/tasks/*` по
  алгоритму `hw.js` (поиск темы по убывающим префиксам id, `findProto`,
  интерполяция `${param}`, слияние `defaults`+`answer_spec`). Все 3561
  прототипов имеют явный ответ — интерпретатор выражений не понадобился.
- Портирована формула прогноза ЕГЭ из `picker_stats.js` (подтема% =
  `subtopic_last3_avg_pct` → секция% = среднее → первичные → вторичные по
  таблице с интерполяцией). **Сходится с вебом до сотых**: на тестовом ученике
  и веб, и приложение показывают «16 из 100, первичные 2,84, +54 до цели».

### 1.2. Реализованные экраны (SwiftUI, native)

Ученик:
- **Вход** — повторяет `auth.html` (логотип, карточка, email/пароль, показ пароля);
- **Главная** — прогноз ЕГЭ (карточка с гейджем и целью 70), «Выбрать/Сбросить все»,
  «Перемешать задачи», аккордеон 12 секций × 84 подтем с %-полосками точности,
  покрытием и счётчиками «− N +», бар «Начать тренировку»;
- **Тренировка** — прохождение, проверка ответов (порт `checkFree`/`normalize`),
  подтверждение при пустых ответах, отчёт с разбором, запись в статистику
  (`write_answer_events_v1`);
- **Мои ДЗ** — счётчики несданных/всего, карточки с бейджами «Сдано/Не сдано»,
  «верно N из M» (добирается из попытки, как в вебе);
- **Выполнение ДЗ** — полный флоу `hw.js`: токен → доступ → попытка → задачи
  (текст+SVG-рисунок+поле «Ответ») → «Не заполнено N из M. Сдать домашнее
  задание?» → сдача → «ДЗ сдано! Верно N из M. Точность P%»;
- **Результат/разбор** — итоговые плашки, верно/неверно, ваш ответ, правильный
  ответ, кнопка «Видео-решение» (карта Rutube из `content/video/rutube_map.json`
  с фоллбэком по base-id);
- **Статистика** — период (7/14/30/90) и источник (всё/ДЗ/тренировка), метрики
  «Последние 10 / период / всё время», покрытие, «Что тренировать сейчас»
  (слабые подтемы), темы с раскрытием подтем;
- **Профиль** — данные, входящие запросы преподавателей (принять/отклонить),
  «Мои преподаватели» с отключением доступа (consent-модель), выход.

Преподаватель:
- **Главная (подбор)** — выбор ученика, карточка прогноза (плейсхолдеры «—» до
  выбора — как в вебе), фильтры (нерешённое/давно не решал/нестабильно/слабые
  места) со счётчиками, аккордеон тем `teacher_picking_screen_v2` с полосками,
  покрытием и бейджами («не решал», «давно не решал», «нестабильно», «слабое»),
  счётчики задач, бар «Создать ДЗ»;
- **Мои ученики** — приглашение по email (consent, человекочитаемые ошибки),
  pending-запросы с отменой, карточки учеников с метриками
  (`teacher_students_summary`), отвязка;
- **Карточка ученика** — шапка, выполненные работы (`list_student_attempts`) с
  переходом к результату, переход к полной статистике (teacher-scope);
- **Создание ДЗ** — название (дефолт «ДЗ DD.MM»), подбор задач по выбранным
  темам (`teacher_picking_resolve_batch_v1`) с превью условий и удалением,
  создание (insert `homeworks` + `homework_links` + RPC `assign_homework_to_student`),
  success-блок «ДЗ создано и назначено…» со ссылкой (копирование/share);
- **Просмотр результата ученика** — `get_homework_attempt_for_teacher` + условия
  задач из контента, ответы/правильные ответы/итог.

### 1.3. Сервисный слой

`SupabaseClient` (actor, URLSession): GoTrue password-login/refresh/logout,
PostgREST RPC/select/insert, авто-refresh + повтор на 401, сетевые ретраи только
на сбоях сети (350/800/1500 мс — паритет с `fetchWithRetry` веба), сессия в
Keychain (восстанавливается после перезапуска). Поверх — `AuthService`,
`HomeworkService`, `StudentService` (аналитика+consent ученика+запись попыток),
`TeacherService` (ученики/подбор/создание ДЗ), `ContentService` (каталог,
манифесты, сборка вопросов, видео-карта), `AnswerChecker`, `ScoreForecast`.
Ошибки всюду человекочитаемые (нет сети/таймаут/неверный пароль/нет доступа) —
raw JSON наружу не показывается.

### 1.4. Верификация (без Xcode)

- Все 38 Swift-файлов приложения проходят `swiftc -typecheck` (macOS SDK,
  iOS-специфика изолирована `#if os(iOS)`).
- **Интеграционный прогон против продакшена: 42/42 проверок зелёные**
  (`DevHarness/main.swift`): юнит-тесты проверки ответов и прогноза; неверный
  пароль; логин/refresh/профиль обеих ролей; список ДЗ; ДЗ по токену; сборка
  текста задач с прод-контента; результат попытки; аналитика self/teacher;
  прогноз; consent-списки; каталог; клиентский подбор; picking screen; работы
  ученика; отчёт для учителя; resolve-батч; **сдача ДЗ учеником (write)**;
  **создание+назначение ДЗ учителем (write)** и его видимость учеником;
  отказ доступа по чужому `student_id`.

## 2. Что работает полностью (проверено против продакшена)

- **Auth**: вход по email/паролю, человекочитаемые ошибки, refresh-токен,
  сессия в Keychain (сценарий C), logout, роутинг по роли из `profiles`.
- **Ученик**: главная с прогнозом (паритет с вебом до сотых), список ДЗ,
  открытие ДЗ, полный цикл сдачи (start → ответы → подтверждение → submit →
  итог; сдано реальное QA-ДЗ «W13-С1 0429», верно 1 из 1), просмотр результата
  с правильными ответами и видео-ссылкой, статистика с фильтрами, тренировка
  с записью в `answer_events`, consent (мои преподаватели/входящие запросы).
- **Преподаватель**: список/метрики учеников, pending-запросы (чтение, отмена,
  приглашение), picking-экран с фильтрами и бейджами, аналитика ученика,
  выполненные работы, отчёт по попытке, resolve-подбор, **полный цикл создания
  ДЗ** (создано и назначено реальное ДЗ, ученик его видит).
- **Ошибки**: нет сети (ретраи+понятный текст), неверный пароль, нет доступа
  (STUDENT_NOT_LINKED → «Нет доступа»), пустые списки (empty-states).
- **Контент**: каталог, тексты задач, SVG-рисунки, проверка ответов
  (ege_decimal/number/дроби/нормализация), карта видео-решений.

## 3. Что работает частично

- **Тренировка ученика**: подбор задач — случайные прототипы выбранных подтем
  (клиентский). Веб использует pick-движок с весами/анти-повторами и self-гейт
  teacher-RPC для фильтров — не портировано. Сценарий «выбрал темы → решил →
  результат в статистике» работает.
- **Фильтры подбора у учителя**: применяются к RPC (счётчики и состав тем
  меняются), но точечный выбор прототипов внутри подтемы (модалка прототипов
  веба) не реализован — выбор на уровне подтем.
- **Видео-разбор**: внешняя ссылка на Rutube, не встроенный плеер.
- **Прогноз на главной учителя**: метрики и полоски — из аналитики (как в
  вебе), но «термометра» в шапке нет.
- **Скриншоты приложения**: сняты на macOS-рендере, не в iOS Simulator (§4.1).

## 4. Что не удалось / не делалось

### 4.1. Главный блокер: на машине нет Xcode

Установлены только Command Line Tools (нет iOS SDK, `xcodebuild`, `simctl`,
симуляторов). Установить Xcode автономно нельзя (App Store/Apple ID).
Следствия:
- приложение не запускалось в iOS Simulator;
- скриншоты приложения — macOS-рендер реальных вью, не Simulator;
- сборка именно `xcodebuild`'ом не проверялась (код проверен `swiftc`-typecheck
  + интеграционно; `project.pbxproj` написан в современном формате Xcode 16
  с synchronized folders — файлы подхватываются автоматически, без ручных
  ссылок, что минимизирует риск ошибок проекта).

**Что нужно от владельца**: открыть проект в Xcode 16+, Run на симуляторе.
Ожидаемые правки, если что-то не соберётся — точечные (iOS-модификаторы),
оценка: минуты–час, ядро проверено.

### 4.2. Не делалось осознанно (вне P0 или только веб)

- Регистрация/сброс пароля/Google OAuth (PKCE-редирект требует настройки
  redirect-URL в Supabase — отметил как ручную задачу владельцу; вход по
  паролю работает);
- session-ссылки (`?session=`), архив ДЗ (счётчик показывается, список — в вебе),
  «Решить аналог», умная тренировка (smart-режим), рисовалка-оверлей, печать,
  редактирование профиля из приложения (RPC-обёртка `update_my_profile` готова,
  UI не выводил — в вебе);
- iPad-раскладки (target — iPhone), тёмная тема.

## 6. Как запустить

См. `README.md`. Кратко: Xcode 16+ → открыть `EGETrainerApp.xcodeproj` →
iPhone-симулятор → Run → войти тестовым аккаунтом из `.env.local`.

## 7. Безопасность ключей

- service_role key — **не использовался** (и не известен приложению);
- database password — **не использовался**;
- использован только **публичный anon-ключ** — тот же, что вшит в клиентский
  JS сайта (`app/config.js`), + JWT пользователей после логина;
- секреты в репозиторий **не добавлялись**; пароли тестовых аккаунтов в коде
  и отчётах не фигурируют (читаются из env / вводятся руками);
- доступ к данным — через те же RLS/SECURITY DEFINER RPC, что у веба.
