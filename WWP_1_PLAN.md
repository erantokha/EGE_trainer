# WWP.1 — Web performance: быстрые вторичные экраны

## 1. Контекст

После ускорения подбора задач основная воспринимаемая задержка веб-версии
сместилась на вторичные экраны:

- самостоятельная статистика ученика;
- список учеников и карточка ученика у преподавателя;
- «Мои ДЗ» ученика;
- повторные переходы между MPA-страницами, где заново загружается каталог;
- общий header, который на холодном старте отдельно читает имя и роль профиля.

## 2. Цель волны

Сократить критический путь первой отрисовки данных и сделать повторные переходы
между страницами практически мгновенными при наличии свежего кеша.

## 3. Подтверждённые причины

1. Каталог кешируется только в памяти JS-модуля и теряется при каждом MPA-переходе.
2. `stats.js` и `student.js` последовательно ждут каталог, а затем аналитику.
3. `my_students.js` ждёт каталог после списка/сводки и блокирующе ждёт pending-заявки.
4. Header делает два независимых запроса к `profiles` за `first_name` и `role`.
5. «Мои ДЗ» ждёт N дополнительных attempt-RPC до первой отрисовки списка.

## 4. Scope реализации

### 4.1 Общий кеш

- Добавить build-scoped JSON-кеш с session/local TTL.
- Подключить persistent-кеш к canonical catalog provider.
- Добавить кеш payload `student_analytics_screen_v1` по viewer/student/days/source.
- При наличии кеша показывать данные сразу, затем обновлять live-ответом.

### 4.2 Критические пути

- `stats.js`: запускать каталог и analytics RPC параллельно; использовать кеш.
- `student.js`: запускать каталог и analytics RPC параллельно; использовать кеш.
- `my_students.js`: загружать каталог вместе со списком и сводкой; использовать
  безопасный кеш роли; pending-заявки не должны блокировать готовность списка.
- `header.js`: объединить чтение имени и роли профиля в один single-flight запрос.
- `my_homeworks.js`: рисовать summary сразу, а недостающие оценки догружать в фоне.
- На странице «Мои ДЗ» не запускать дублирующий summary-запрос ради колокольчика.

## 5. Не входит в волну

- Изменение или выкладка SQL/RPC-контрактов Supabase.
- Полное устранение attempt-RPC из backend-потока «Мои ДЗ».
  В этой волне они удаляются из критического пути первого рендера.
- Изменение продуктовой логики, состава статистики и прав доступа.

## 6. Риски и защита

- Кеш всегда изолирован build-id и ключом пользователя/ученика.
- Кеш используется только как stale-while-revalidate; live RPC остаётся источником истины.
- Ошибка фонового обновления не должна стирать уже показанные данные.
- Проверка роли из кеша не заменяет backend authorization/RLS.

## 7. Проверки

1. Governance:
   - `node tools/check_runtime_rpc_registry.mjs`
   - `node tools/check_runtime_catalog_reads.mjs`
   - `node tools/check_no_eval.mjs`
2. Unit/syntax:
   - профильные unit-тесты и `node --check` изменённых модулей.
3. Playwright smoke:
   - ученическая главная/статистика;
   - teacher screen prewarm;
   - вторичные performance-сценарии.
4. Build/cache:
   - `node tools/bump_build.mjs`
   - проверка согласованности build-id.

## 8. Definition of Done

- Повторный MPA-переход не требует повторного catalog RPC в пределах TTL/build.
- Статистика и карточка ученика не сериализуют каталог и analytics RPC.
- Список учеников отображается без ожидания pending-заявок.
- Header делает не более одного собственного запроса к `profiles` на cache miss.
- «Мои ДЗ» отображает summary до завершения attempt-RPC.
- Все обязательные проверки зелёные, изменения закоммичены и отправлены в GitHub.
