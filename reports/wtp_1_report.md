# WTP.1 — отчёт по прогреву teacher-home

Дата: 2026-06-13.

## Корень задержки

После `list_my_students` веб наполнял список, но не загружал
`teacher_picking_screen_v2(mode='init')`. Именно этот payload питает аккордеон,
бейджи, прогноз и рекомендации, поэтому первый выбор ученика всегда ждал cold RPC.
Существующий snapshot-прогрев относился только к подбору задач.

## Реализация

- После успешного списка запускается прогрев первых `min(10, N)` учеников.
- Фоновый прогрев ограничен двумя параллельными screen RPC.
- Добавлен общий single-flight кеш по `studentId + filterId`, TTL 60 секунд,
  stale-while-revalidate и лимит 24 записи.
- Первый выбор прогретого ученика синхронно применяет payload без ожидания сети.
- Незавершённый прогрев переиспользуется ручным выбором.
- Сохранённый фильтр восстанавливается до запуска прогрева.
- Ошибки фонового прогрева не блокируют список и cold-fetch выбранного ученика.

## Проверки

- Новый mobile-viewport e2e проверяет:
  - прогреваются только первые `min(10, N)`;
  - фоновых init RPC одновременно не больше двух;
  - прогрев учитывает сохранённый `weak_spots`;
  - первый выбор не создаёт повторный init RPC;
  - экран готов менее чем за одну секунду.
- Teacher characterization аккордеона зелёный.
- Teacher browser smoke приведён к canonical v2 contract:
  `weak_spots` поддерживается, `selection.normalized` опционален.
- Smoke helper теперь ждёт финальный статус, а не подстроку `OK` внутри `smoke`.
- Weak-spots e2e учитывает canonical local-resolve из прогретого snapshot и
  сетевой fallback, не требуя лишнего resolve RPC.
- Финальный последовательный teacher e2e-набор: `6 passed`.
- Governance: runtime RPC registry, catalog reads и no-eval зелёные.
- Синтаксические проверки и `git diff --check` зелёные.
