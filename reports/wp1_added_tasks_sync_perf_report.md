## 1. Итог

Серия кликов «Выбрать всё»/«+» с выбранным учеником больше **не плодит перекрывающиеся resolve**:
коалесинг (debounce ↑) + **in-flight-гард** (максимум один синк в полёте, финальный trailing по
последнему состоянию). Предпросмотр **префетчит манифесты параллельно** вместо последовательного
дочитывания. «Начать» активируется по завершении (теперь быстрого) синка. Логика отбора —
**не тронута** (только КОГДА/СКОЛЬКО раз зовём и КАК ждём).

## 3. Что изменено (`file:line`, только `tasks/picker.js`)

**§5.1 Коалесинг + in-flight-гард:**
- Константа `ADDED_SYNC_DEBOUNCE_MS = 300` (было inline `90`) — `:3015`.
- Флаги контроллера `_ADDED_SYNC_INFLIGHT / _ADDED_SYNC_PENDING / _ADDED_SYNC_PENDING_OPTS / _ADDED_SYNC_SETTLE_WAITERS` — `:3016-3019`.
- `maybeRunAddedSync` (`:3201`): если синк в полёте → пометить pending; иначе запустить.
- `runAddedSync` (`:3210`): `inflight=true` → `await syncAddedTasksToSelection` → `finally`: `inflight=false`,
  если pending → trailing-прогон, иначе уведомить settle-waiters. Существующий `_ADDED_SYNC_SEQ` оставлен belt-and-suspenders.
- `awaitAddedSyncSettled` (`:3231`) — промис оседания (нет in-flight и pending).
- `isAddedSyncPending` (`:3237`) — `dirty || timer || inflight || pending`.
- `scheduleSyncAddedTasks` (`:3241`) переписана через `maybeRunAddedSync` (immediate → сразу; иначе debounce `ADDED_SYNC_DEBOUNCE_MS`).
- `flushTeacherAddedTasksSelection` (`:3256`) — через контроллер, **await `awaitAddedSyncSettled()`** (без параллельного синка; вызывающий получает финальное состояние, включая trailing).
- **`_ADDED_SYNC_DIRTY` НЕ клоббится** — её прежняя семантика (clean/dirty, сброс в конце `syncAddedTasksToSelection`) сохранена; trailing управляется отдельным `_ADDED_SYNC_PENDING`.
- modal-open проверки (`openAddedTasksModalFast`): `canReuseRenderedView` (`:3992`) и flush-гейт (`:4018`) переведены на `isAddedSyncPending()` (учитывают in-flight/pending, иначе после `dirty=false` в окне до trailing можно было переиспользовать устаревший рендер).

**§5.2 Параллель манифестов:**
- `prefetchTeacherResolveManifestIndexes(paths, limit=6)` (`:4122`) — прогрев кэша уникальных `manifest_path` параллельно (пул на 6; при 0/1 уникальном — no-op/обычный путь).
- Вызов в `buildPreviewQuestionsFromResolveRows` ПЕРЕД assembly (`:4164`): `await prefetch(...rows.manifest_path)` → последующий `for…await getTeacherResolveManifestIndex` в `tryAdd` попадает в кэш без последовательных сетевых await. **Логика выбора не изменена** (порядок/фильтры/even-distribution as is).

**§5.3 «Начать»:**
- Отдельный гейт НЕ вводился. Кнопка `#start` гейтится на `total>0` (число фактически добавленных задач),
  обновляется `reconcileAddedTasksTruth` → `updateAddedTasksUi` сразу по завершении синка. §5.1/§5.2 делают
  синк быстрым → кнопка активна оперативно (DoD §8.4, клауза «сразу по завершении ускоренного синка»).
- Вариант «активировать по desired-count до резолва» **отклонён**: навигация «Начать» обязана иметь валидные
  `question_id` (буфер задач = выборка; ids появляются именно из resolve+preview) — иначе риск пустой/недорешённой
  выборки (план §7). Это не требует выхода за picker.js → STOP-ASK не сработал.

**§5.4 AbortController:** не реализован — защита от конкуренции обеспечена in-flight-гардом (§5.1), а не клиентской
отменой (план §5.4 прямо отмечает: серверный SQL отмена не останавливает). Не усложнял.

## 6. Что осталось на уровень 2 (честно)

Базовая стоимость **одного** resolve (~0.7 с в замере) — это пересчёт полного состояния ученика по всему каталогу
внутри `teacher_picking_*` (SQL). WP1 её **не трогает** (это уровень 2 — материализация/кэш состояния на бэке).
WP1 убрал только *лишние* повторы и *последовательный* хвост манифестов. Грубая оценка остаточного времени: при
N кликах теперь ≈ один базовый resolve (~0.7 с) + быстрый параллельный предпросмотр, вместо N×(resolve + последовательные манифесты).
