# Аудит производительности: подбор задач с фильтром (~6с)

> Анализ без правок кода. Запрос: при активном фильтре предпросмотр/старт ждут ответ Supabase ~6с; цель 2–3с, идеал <1с.

## Измеренный корень (Playwright, тест-аккаунт ученика)

Сценарий «фильтр `weak_spots` + Выбрать всё» → до готовности «Предпросмотр» **5.4с** (на аккаунте с реальной историей — ~6с). Рендер модалки после клика — **54мс** (клиент НЕ узкое место). Сетевой ответ RPC — **3KB** (сеть/объём НЕ при чём).

```
до готовности «Предпросмотр»: 5379мс | рендер модалки: 54мс
resolve-RPC: 12 шт, сумма 15353мс, макс 1700мс  (каждый ~0.9–1.7с)
все 12 = teacher_picking_resolve_batch_v1, 200 OK
```

**Формула 6с = два множителя:**
1. **Клиентский веер:** «Выбрать всё» дробится на **12 отдельных RPC** (по одному на секцию), concurrency cap = **4** → **3 волны**.
2. **Медленный SQL:** каждый вызов **~1.5с серверного compute**.
   Итог wall-time ≈ 3 волны × ~1.7с (худший в волне) ≈ **5.4с**.

## Почему каждый вызов ~1.5с (серверный SQL)

RPC `teacher_picking_resolve_batch_v1` / `teacher_picking_screen_v2` на каждый вызов:

1. **`answer_events` (история попыток) сканируется ~4 раза за вызов:**
   - `student_proto_state_v1` зовётся ДВАЖДЫ (как `proto_state` и внутри `student_topic_state_v1`), planner не мемоизирует разные функции;
   - внутри `student_proto_state_v1` — ДВА полных скана (`proto_events` all-time + `proto_last3` с оконной сортировкой `row_number() over (partition by unic_id order by ... desc)`).
2. **Нет индекса под чтение** `answer_events(student_id, …)` — в репозитории нет ни одного `CREATE INDEX` для этой таблицы; при росте общей таблицы это seq-scan на каждый из 4 проходов.
3. **Градиент считается по ВСЕМ прототипам каталога** независимо от scope (даже подбор в одной подтеме поднимает all-time состояние всех ~сотен прототипов; фильтр применяется `WHERE` уже поверх полного `candidate_base`).
4. **`student_proto_state_v1` — `language plpgsql`** → planner НЕ инлайнит её и не проталкивает scope-предикат внутрь скана (подтверждено внешними источниками: plpgsql/STABLE не инлайнятся).
5. `proto_last3` (оконная сортировка) нужен только для бейджа «%/3», не для отбора фильтром, но считается всегда.

## Почему 12 вызовов (клиентский веер)

`tasks/picker.js`: «Выбрать всё» (все секции с одинаковой дельтой K, путь `uniformK`) идёт **по одному RPC на секцию** через `runCapped(secEntries, 4, …)` — это осознанный обход серверного таймаута/`concurrency-500` (один батч на 12 секций таймаутил). Cap = 4 захардкожен. Плюс:
- дебаунс префетча предпросмотра **500мс**;
- «Начать» (`#start`) активна по `total>0` сразу, но `saveSelectionAndGo` делает `await resolve` на клике → на холодном кэше все 12 RPC оплачиваются ПОСЛЕ клика, без индикации;
- кэш `STUDENT_RESOLVE` корректно переиспользуется preview↔start (двойного резолва нет) — проблема в ТАЙМИНГЕ прогрева, а не в дублировании.

## Варианты оптимизации с оценкой выигрыша

### Серверные (уменьшают ~1.5с на вызов) — главный рычаг

| # | Что | Выигрыш | Риск/сложность |
|---|-----|---------|----------------|
| **S1** | **Индекс** `answer_events(student_id, question_id)` (covering: + `correct, occurred_at, created_at, source`), создавать `CONCURRENTLY` | если сейчас seq-scan → **×3–10 на сканах, −3…−5с** с вызова | низкий риск, 1 DDL (red-zone: горячая таблица) |
| **S2** | **Убрать двойной вызов** `student_proto_state_v1` (считать proto-state 1 раз, topic-агрегаты выводить `group by` из него) | **×~2, −2…−3с** | средний (сохранить семантику topic-mastery) |
| **S3** | **Убрать `proto_last3`** из горячего пути (он только для бейджа) | −1 скан + −1 сортировка, **−1…−2с** | низкий-средний (shared layer-3, нужен смоук) |
| **S4** | **Сузить scope:** протолкнуть фильтр секции/темы в `visible_questions` ДО скана `answer_events` (требует перевода функции на `language sql` для инлайна) | для узких scope **×N (N=число секций)**; для `global_all` — 0 | средний-высокий |
| **S5** | **Денормализация:** сводная таблица `student_proto_agg` (attempt/correct/last_attempt/last3) на триггере insert в `answer_events` → resolve читает готовое, без скана истории | **−4…−5с, подбор «мгновенный»** | высокий (write-path/триггеры, red-zone) |
| **S6** | **Materialized view** proto-state + `REFRESH CONCURRENTLY` после submit/по расписанию | **−4…−5с** | средний (свежесть данных) |

### Клиентские (уменьшают «12 вызовов / 3 волны»)

| # | Что | Выигрыш | Риск |
|---|-----|---------|------|
| **C1** | **Один батч-RPC на все 12 секций** вместо 12 (BE-функция принимает массив `p_requests`) — станет возможным, если S1–S3 уберут серверный таймаут | 12→1 раундтрип: **5.4с → ~1.5–2с** (а с S1 → **<1с**) | средний (вернуть то, что раньше таймаутило — только ПОСЛЕ серверного ускорения) |
| **C2** | **Поднять concurrency cap 4→8/12** (если C1 пока нельзя) | 3 волны → 1–2 волны: **5.4с → ~1.8–3.4с** | низкий-средний (проверить лимит пула Supabase) |
| **C3** | **Немедленный prewarm** (без дебаунса 500мс) на bulk-действиях + «Начать» ждёт тёплый кэш с индикатором | убирает повторную/запоздалую оплату на старте: клик «Начать» становится ≈мгновенным (резолв уже в фоне); −0.5с + UX | низкий |
| **C4** | **Снизить over-fetch** (`overN = delta + 6`, cap 40) для bulk | меньше серверной работы на каждый из 12 → каждый вызов чуть быстрее | низкий |

### Диагностика (0 риска, сделать первым)
`EXPLAIN (ANALYZE, BUFFERS)` на live для `teacher_picking_resolve_batch_v1(...)` и `student_proto_state_v1(uuid,'all')`; `SELECT count(*) FROM answer_events WHERE student_id=…`; `SELECT indexdef FROM pg_indexes WHERE tablename='answer_events'` — подтвердить seq-scan и число повторов функций.

## Рекомендованный план и итоговая оценка
1. Диагностика (EXPLAIN) — 5 мин.
2. **S1 (индекс)** — самый дешёвый крупный выигрыш (вызов ~1.5с → ~0.3–0.5с).
3. **S2+S3** (дедуп сканов) — ещё ×~2.
4. После серверного ускорения — **C1** (12 вызовов → 1 батч) или **C2** (cap↑).
5. **C3** (prewarm/UX) — параллельно, дёшево.

**Прогноз:** S1 + (C2 или C1) → **~6с → ~1.5–2с**. S1 + S2/S3 + C1 (один быстрый батч) → **<1с**. S5/S6 (денормализация/matview) — стратегический путь к стабильному <1с при росте истории.

**Ключевой вывод:** узкое место — НЕ сеть и НЕ клиент-рендер, а серверный compute, помноженный на дробление «Выбрать всё» на 12 запросов. Бить нужно по обоим: индекс+дедуп сканов на сервере и схлопывание 12 вызовов в 1 (или больше параллелизма) на клиенте.
