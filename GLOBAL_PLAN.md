Дата обновления: 2026-04-22
Репозиторий: `EGE_rep`

## 1. Назначение документа

Этот файл фиксирует не прошлую миграцию, а текущую рабочую roadmap проекта после завершения перехода на 4-layer архитектуру.

Документ отвечает на вопросы:
- что сейчас главное по приоритету
- какие волны логично делать следующими
- что даёт каждая волна продукту
- что не нужно смешивать в одну инициативу

Связанные документы:
- `PROJECT_STATUS.md`
- `docs/navigation/current_dev_context.md`
- `docs/navigation/architecture_contract_4layer.md`
- `docs/supabase/runtime_rpc_registry.md`

## 2. Текущее положение

Что уже закрыто:
- migration track Stages 0–10
- canonical read/write seams
- teacher-picking v2
- student analytics v1
- runtime governance baseline

Что теперь является основным фронтом работ:
- стабилизация screen/print layout
- развитие продуктовых ролей и сценариев
- снижение сложности frontend orchestration
- поддержание инфраструктуры управления проектом в актуальном состоянии

## 3. Приоритеты верхнего уровня

### P0

- восстановить стабильный mobile screen layout на `trainer/list/unique`
- развести screen-layout и print-layout по отдельным режимам

### P1

- поддерживать опорные управляющие документы и процесс волн
- использовать процессную роль куратора как planning-layer/gateway
- определить roadmap по новой роли без разрушения teacher/student модели
- teacher smoke / teacher visual e2e временно отложить до завершения
  screen/print стабилизации

### P2

- снизить сложность крупных frontend-модулей
- оценить переход на сборку как отдельный техтрек
- улучшить тестируемость ключевых product flows

### W2 — Разводка screen-layout и print-layout

Статус: 🔄 в процессе
Приоритет: P0

Цель:
- развести экранную и печатную вёрстку карточек задач по отдельным режимам

Контекст:
- print-related layout-правила протекли в screen CSS и уже дали mobile regression

Основные затронутые зоны:
- `tasks/trainer.css`
- `tasks/list.js`
- `tasks/unique.js`
- `app/ui/print_btn.js`

Ожидаемый результат:
- screen layout не зависит от print-layout
- print-layout включается только в явном print-state
- mobile `trainer/list/unique` стабилен

Фактическое состояние по репозиторию на `2026-04-22`:
- `W2.0` завершена: baseline и карта конфликтующих правил зафиксированы в `docs/navigation/print_layout_inventory.md`
- `W2.1` завершена: введён managed print-state через `body.print-layout-active`, lifecycle вынесен в `app/ui/print_lifecycle.js`, page-level hooks переведены на общий контур
- `W2.2` завершена по коду: в `tasks/trainer.css` screen-layout очищен от основных print-driven компромиссов для `vectors/derivatives/graphs`, введён отдельный mobile stacking для figure-cards
- `W2.3` завершена по коду и документам: print-layout собран в явный логический контур внутри `@media print`, а layout-селекторы привязаны к `body.print-layout-active`; контракт зафиксирован в `docs/navigation/print_layout_contract.md`, отчёт — в `w2_3_report.md`
- `W2.4` завершена по повторному evidence: после кураторского отклонения из-за mobile-regression добавлены geometry assertions, исправлена ширина mobile trainer, повторно зелёные `visual-walkthrough` и `w2-4-print-layout`; отчёт — в `w2_4_report.md`
- `W2.6` завершена и принята: первичный acceptance из `w2_6_report.md` дополнен follow-up пакетом `w2_6_fix_report.md`, после чего review-cycle закрыт без нового planning split
- `W2.5` не начата: структурное закрепление CSS по ответственности отложено до стабилизации поведения

Текущие работы внутри активной волны `W2`:
- следующая подволна: `W2.5`
- цель следующего прохода: структурно закрепить CSS screen/print по ответственности после завершённого acceptance, не перепридумывая уже принятый `W2.6`
- основной пакет работ:
  - разнести CSS-ответственности между screen-layout, print-layout и page-specific overrides
  - убрать наиболее опасные смешанные правила, из-за которых следующий print-fix может снова сломать mobile screen
  - зафиксировать структуру без отката уже принятых контрактов `W2.6`
  - собрать узкий evidence packet по post-stabilization cleanup
- что не смешиваем с текущим проходом:
  - повторный acceptance `W2.6`, если не выявлен новый дефект
  - отдельную стабилизацию `setup-student` flake, если она не блокирует новый CSS cleanup
  - любые product/backend изменения вне screen/print acceptance

Текущий критический путь:

`W2.5`

Что уже сделано на критическом пути:
- baseline инвентаризации собран
- print-state и cleanup уже централизованы
- screen-layout карточек очищен от основных print-driven компромиссов в `tasks/trainer.css`
- print-layout собран в отдельный state-gated контур поверх `@media print`
- появился Playwright baseline и подтверждённый visual walkthrough для student-flow
- W2.4 visual spec покрыл desktop trainer, mobile trainer, unique screen, print без ответов/с ответами и ключевые figure cases
- mobile trainer width regression исправлена; W2.4 повторно прошла `visual-walkthrough` и `w2-4-print-layout`

Почему критический путь теперь выглядит так:
- `W2.0` и `W2.1` уже дали безопасную опору, их повторно планировать не нужно
- `W2.2` уже вернула экранный layout под отдельный control layer, но это ещё не означает завершение всего W2
- `W2.3` уже дал отдельный print-контур, поэтому возвращаться к нему как к следующей задаче не нужно
- `W2.4` уже закрыл критичный feedback по mobile trainer и закрепил geometry assertions
- `W2.6` уже полностью закрыла acceptance-layer по `trainer/list/unique`, поэтому возвращаться к ней как к активной подволне не нужно
- `W2.5` теперь становится единственным шагом до устойчивого screen/print режима

Ближайшие следующие шаги без перепридумывания roadmap:
- открыть и выполнить `W2.5` как post-stabilization cleanup
- не смешивать с `W2.5` новые product/layout инициативы, если не найден отдельный дефект
- отдельно не смешивать с `W2.5` стабилизацию известного `setup-student` session-capture flake, если он не блокирует CSS cleanup
- после `W2.5` переходить к следующим продуктовым направлениям без возврата к уже принятому acceptance-пакету

Что считать завершением текущих работ по волне:
- `W2.5` получает собственный отчётный артефакт и фиксирует структуру CSS по ответственности;
- уже принятый acceptance по `trainer/list/unique` не регресснут;
- по итогам критический путь по screen/print считается закрытым, либо фиксируется отдельный follow-up, если cleanup выявит новый системный хвост.

## 8. Формат волн

Каждая новая волна должна иметь:
- цель
- контекст
- out of scope
- затрагиваемые файлы
- пошаговый план
- риски / stop-ask точки
- DoD
- план проверки
- отчётный артефакт

Если волна большая или архитектурно спорная:
- выносить её в отдельный `*_PLAN.md`
