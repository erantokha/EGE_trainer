# WIOS_1_PLAN — iOS-приложение: полный функциональный паритет с сайтом

Дата: 2026-06-11. Куратор: Claude (роль принята по CLAUDE.md/CURATOR.md).
Оператор утвердил скоуп в чате 2026-06-11 (см. §3 про вычеркнутое).
Статус: утверждён оператором, исполнение — автономное в этой же сессии.

---

## 1. Цель

Довести `ios/EGETrainerApp` до функционального паритета с веб-версией
тренажёра для обеих ролей (ученик/учитель): все user-facing фичи сайта,
кроме явно вынесенных за скоуп (§3), работают в приложении на iPhone.

## 2. Контекст и мотивация

Ночной автономный прогон 2026-06-11 построил P0-версию приложения
(вход, тренировка, ДЗ, статистика, подбор и создание ДЗ учителем,
consent) — 42/42 интеграционных проверок против прода. Однако задача
оператора была «полностью повторить функционал», и при gap-анализе
2026-06-11 (3 разведки: фичи ученика, фичи учителя, состояние iOS)
найдено ~25 пропусков, включая не отмеченные в REPORT.md:
рендер LaTeX-формул (контент содержит `\frac`/`\sqrt`, сайт рендерит
MathJax), пошаговый режим «Тестирование» с таймером, фильтры подбора
на главной ученика. Xcode 26.3 установлен, среда рабочая, приложение
запускается в Simulator (iPhone 17, iOS 26.3.1).

## 3. Out of scope (зафиксировано оператором)

- «Умное ДЗ» (рекомендации тем) и «Вариант 12» на карточке ученика —
  оператор: убраны и из веб-тренажёра, не переносить.
- iPad-раскладки, Sentry/крэш-мониторинг.
- Universal links (открытие веб-ссылок В приложении) — требует платный
  Apple Developer аккаунт; генерация/шеринг ссылок ИЗ приложения — в скоупе.
- Печать на бумагу как отдельный флоу — только PDF-экспорт + системный
  share (AirPrint доступен из share sheet бесплатно).
- TestFlight / App Store / распространение.
- Любые правки веб-кода, SQL, контрактов — приложение подстраивается
  под существующие контракты, не наоборот.
- Live-тест Google OAuth, регистрации и сброса пароля — после волны
  (требуют действий оператора: redirect-URL в Supabase, коды из почты).
  Код этих фич — в скоупе волны, приёмка кода — статическая + негативные
  сценарии.

## 4. Затрагиваемые файлы

Все изменения — строго внутри `ios/EGETrainerApp/**` и `reports/**`:

- `EGETrainerApp/Services/*` — расширение: SupabaseClient (signup,
  recover, OAuth token exchange), AuthService, StudentService
  (фильтры/спред подбора), TeacherService (прото-статистика, session
  links), ContentService (аналоги, превью), новые PDFExporter,
  SessionDraftStore.
- `EGETrainerApp/Models/*` — новые модели (архив ДЗ, прото-статистика,
  session link, черновик сессии).
- `EGETrainerApp/Screens/**` — правки всех экранов + новые: пошаговый
  тренажёр, аналог, архив ДЗ, модалка прототипов, предпросмотр
  добавленных, просмотр подборки, регистрация/сброс, редактирование
  профиля.
- `EGETrainerApp/DesignSystem/*` — MathTextView (WKWebView+KaTeX/MathJax),
  RutubePlayerView, InfoPopover (тултипы), адаптивные цвета (тёмная тема),
  DrawOverlay (PencilKit).
- `EGETrainerApp/App/*` — deep-link scheme `egetrainer://` (Info.plist
  в pbxproj-настройках), роутинг auth-callback.
- Vendored ресурсы: локальный KaTeX или MathJax-минимум в bundle.
- `DevHarness/main.swift` — новые интеграционные проверки.
- `reports/wios_1_report.md` + `reports/wios_1/*` (скриншоты).
- Куратор после приёмки: `GLOBAL_PLAN.md`, `PROJECT_STATUS.md` (sync).

Веб-код (`tasks/*`, `app/*`), `docs/supabase/*`, `tools/*`,
`content/*` — **read-only** (источник логики для портирования).

## 5. Пошаговый план

> **Task-tracking (обязательно для исполнителя):** в начале работы
> создай TaskList через инструмент `TaskCreate` с пунктами §5.1–§5.8
> этого плана. По мере выполнения обновляй статус каждого пункта через
> `TaskUpdate`: `in_progress` при старте шага, `completed` при его
> завершении. Это нужно, чтобы оператор мог наблюдать прогресс в
> реальном времени через task-panel.

### §5.1 Подготовка
Baseline-сборка (`xcodebuild build`, симулятор iPhone 17), фиксация
текущего поведения. Работа без коммитов (коммит/пуш — оператор).

### §5.2 Foundation (кросс-режущие модули)
a. **MathTextView** — рендер LaTeX в стемах/ответах (WKWebView +
   vendored KaTeX с MathJax-совместимыми разделителями как на сайте;
   офлайн, без CDN). Внедрить во все места показа условий: карточки
   вопроса/разбора, превью прототипов, PDF.
b. **Тёмная тема** — адаптивные палитры в Theme.swift (системная тема
   iOS, как ручной toggle на сайте), аудит всех экранов на хардкод цветов.
c. **Тултипы метрик** — InfoPopover + перенос текстов из
   `app/ui/metric_help.js` (прогноз, бейджи, покрытие).
d. **RutubePlayerView** — встроенный плеер (WKWebView embed) вместо
   внешней ссылки в разборе.

### §5.3 Тренировка ученика
a. **Фильтры подбора на главной** (без фильтра/нерешённое/давно/
   нестабильно/слабые) — портировать веб-механику: self-гейт teacher
   RPC + клиентский спред «вширь» по прототипам + «Выбрать всё»
   per-section (источник: `tasks/picker.js` student-ветки, память
   `student-filter-pick-spread`).
b. **Пошаговый режим «Тестирование»** — таймер, прогресс «N/M»,
   вперёд/назад, подтверждение завершения; переключатель режимов
   тест/список на главной (паритет `trainer.html` vs `list.html`).
c. **Отчёт** — фильтр «только неверные», «Новая сессия»/«В меню».
d. **«Решить аналог»** — из карточки разбора: другой прототип той же
   базы (`tasks/analog.js` логика выбора).
e. **Восстановление незавершённой сессии** — persist черновика
   тренировки, предложение продолжить после перезапуска приложения.

### §5.4 ДЗ ученика
a. **Архив ДЗ** — экран с пагинацией по 50
   (`student_my_homeworks_archive`).
b. **Бейдж несданных** на табе «Мои ДЗ».

### §5.5 Учитель
a. **Модалка прототипов** в подтеме — карточки прототипов (превью
   условия через MathTextView), степперы с cap, бейджи точности
   last-3/all-time + дата (`question_stats_for_teacher_v2`,
   `proto_last3_for_teacher_v1`), фоновый прогрев статистики.
b. **Предпросмотр «Добавленные задачи»** на главной — условия, ответы,
   статистика, shortage-подсказка («подобрано 2 из 5»).
c. **«Начать»** — просмотр подборки учителем (лист с ответами).
d. **Создание ДЗ — доводка**: поле «Описание», опция «Не назначать»,
   «Перемешать», добавление задач с экрана создания (модалка
   тем/прототипов), замена задачи.
e. **Session-ссылки** — `create_session_link` (mode list/test) +
   share sheet с подборки.
f. **Поиск-автодополнение** в выборе ученика.
g. **«Термометр» прогноза** — полная карточка как на вебе.
h. **Карточка ученика — доводка**: метрики за периоды, история
   выданных ДЗ со статусами, отвязка с карточки.
i. **«Мои ученики» — доводка**: поиск, фильтр «Проблемные», селекты
   период/источник.

### §5.6 Общие модули
a. **PDF-экспорт** — модуль (HTML-лист → WKWebView → PDF; заголовок,
   с ответами/без) + share sheet. Точки входа: тренировка/список
   ученика, ДЗ, подборка и создание ДЗ учителя (для учителя — вариант
   «с ответами»).
b. **Рисовалка** — PencilKit-оверлей (перо/маркер/ластик/undo,
   нативный инструментарий) на экранах прохождения тренировки/ДЗ и
   просмотра листа. Решение оператора: PencilKit-эквивалент вместо
   точного порта веб-рисовалки.

### §5.7 Auth (red-zone, одобрено оператором 2026-06-11)
a. **Регистрация** — вкладка на AuthView: роль (ученик/учитель), ФИ,
   класс/тип учителя, email/пароль, resend-письмо; GoTrue signup REST
   (как `tasks/auth.js`).
b. **Сброс пароля** — отправка recover-письма из приложения
   (завершение смены — по ссылке в вебе, как договорено).
c. **Google OAuth** — ASWebAuthenticationSession + PKCE, scheme
   `egetrainer://auth-callback`, обмен code→session. Код полностью,
   live-тест — после добавления redirect-URL оператором.
d. **Редактирование профиля** (`update_my_profile`) и **удаление
   аккаунта** (`delete_my_account`, двойное подтверждение).

### §5.8 Приёмка
Полная сборка; прогон DevHarness (старые 42 + новые проверки);
Simulator-прогон обеих ролей по чек-листу; скриншоты всех новых/
изменённых экранов в `reports/wios_1/`; отчёт `reports/wios_1_report.md`.

## 7. Риски и stop-ask точки

- **Red-zone**: §5.7 (auth-flow) — одобрен оператором явно в чате
  2026-06-11; скоуп узкий: только iOS-код, GoTrue REST как у веба,
  без изменений настроек Supabase из кода.
- KaTeX vs MathJax: если KaTeX не покрывает конструкции контента —
  переключиться на vendored MathJax (решение внутри скоупа, отметить
  в отчёте).
- PencilKit в Simulator управляется мышью — приёмка рисовалки в
  Simulator ограничена; отметить в отчёте.
- `several_attempts`/повторные попытки ДЗ: следовать веб-поведению
  `hw.js`, не изобретать.
- Расхождение прод vs репо (фича есть в коде, но выключена на проде) —
  stop-ask.

> **Режим работы: автономный.** Не останавливайся за подтверждением
> на каждом шаге, не спрашивай «продолжать ли», не проси промежуточного
> ревью. Доведи работу до DoD и верни отчёт (`reports/wios_1_report.md`
> + completion summary). Куратор принимает работу целиком по факту.
>
> **Останавливайся (stop-ask) только в следующих случаях:**
> 1. Попытка изменить файл вне §4 «Затрагиваемые файлы».
> 2. Попытка зайти в зону из §3 «Out of scope».
> 3. План противоречит реальности кода (RPC/функции нет, сигнатура
>    не та, контракт разошёлся с реестром).
> 4. DoD объективно недостижим без выхода за scope.
> 5. Governance-скрипт упал и причина не из диффа волны (диффа
>    веб-кода быть не должно вовсе).
> 6. Уязвимость/утечка креденшлов в репозитории.
> 7. Задача распадается на независимые DoD.
> 8. Один сценарий падает 2+ раза подряд, причина неясна.
> 9. Нужно архитектурное решение с влиянием вне §4.
> 10. Проектные триггеры волны: (а) любая потребность изменить
>     веб-код/SQL/контракты; (б) потребность в новом RPC; (в) фича
>     веба не работает на проде (расхождение прод/репо);
>     (г) необходимость платного Apple Developer аккаунта.
>
> **Не экстренное** (решать самому): выбор имён/структуры Swift-кода,
> порядок шагов §5, KaTeX/MathJax выбор, детали UX-адаптации
> веб-паттернов под iOS-конвенции (модалка → sheet, тултип → popover),
> повторные прогоны сборки/тестов.

## 8. Критерии приёмки (DoD)

1. Все пункты §5.2–§5.7 реализованы; ни один не «оформлен как
   follow-up» без stop-ask.
2. Приложение собирается `xcodebuild build` без ошибок и ворнингов
   уровня error; запускается в Simulator (iPhone 17).
3. Формулы контента рендерятся (скриншот темы с `\frac` — например,
   производные 8.1) — не сырой TeX.
4. Для каждой фичи §5 — скриншот из Simulator в `reports/wios_1/`.
5. DevHarness: старые 42 проверки зелёные + новые проверки (архив,
   session link, прото-статистика, signup/recover негативные) зелёные.
6. Сценарии чек-листа §9 пройдены в Simulator вживую.
7. `git status`: изменения только в `ios/**` и `reports/**` (+ план).
8. Отчёт `reports/wios_1_report.md` с фактами и скриншотами.

## 9. План проверки

1. `xcodebuild build -scheme EGETrainerApp -destination 'platform=iOS Simulator,name=iPhone 17'`.
2. DevHarness интеграционный прогон против прода (тестовые аккаунты
   из `.env.local`).
3. Simulator-чек-лист (скриншот на каждый пункт):
   - ученик: вход → главная (фильтр «слабые места» → счётчики
     меняются) → режим «Тестирование» (таймер, N/M, навигация) →
     завершение → отчёт (только неверные) → «Решить аналог» →
     видео-плеер открывается внутри → PDF-экспорт листа → рисовалка
     поверх задачи → Мои ДЗ (бейдж) → архив (пагинация) → профиль
     (редактирование) → тёмная тема (переключение системной);
   - учитель: вход → выбор ученика поиском → модалка прототипов
     (бейджи точности/даты) → предпросмотр добавленных (shortage) →
     «Начать» (лист с ответами) → session-ссылка (share) → создание
     ДЗ (описание, «не назначать», добавление/замена задач) → PDF
     с ответами → карточка ученика (история ДЗ) → «Мои ученики»
     (поиск, «Проблемные»);
   - auth: вкладки регистрации/сброса рендерятся, валидация и
     негативные сценарии (занятый email — человекочитаемая ошибка);
     Google-кнопка открывает OAuth-флоу до экрана Google (дальше —
     post-wave).
4. Кросс-проверка: после iOS-действий открыть веб (Playwright или
   вручную) и убедиться, что созданное ДЗ/попытки видны и корректны.
