## 1. Цель

Полностью убрать мигание бейджей прототипов в модалке подбора (`#protoPickerModal`):
карточки должны появляться **сразу с верными бейджами** (точность X/3 + дата), без
промежуточного ложного «Не решал»/«0/0». Достигается комбинацией **3a**
(рендер карточек только с готовой статистикой) + **3b** (фоновый прогрев кеша,
чтобы на тёплом кеше открытие было мгновенным). Для **обоих ролей** (ученик и
учитель), в одной волне.

## 2. Контекст и мотивация (по аудиту, эта сессия)

Симптом (оператор): при открытии подтемы на «неразогретой» странице бейдж
прототипа сначала показывает «Не решал», потом — реальный. Дезинформирует.

Корень (по коду, `tasks/picker.js`):
- `openProtoPickerModal` (`:2900`) рендерит карточки с **плейсхолдером-бейджем
  «Не решал»** (`renderProtoModalCard` → `setModalStatsBadge(statsBadge, null,
  { emptyLabel: 'Не решал' })`), и только потом — после `await typesetMathIfNeeded`
  (`:2960`) и `await refreshProtoModalBadges` (`:2961`) — заменяет на реальное.
- **R1** плейсхолдер дезинформирует («Не решал» выглядит как настоящий ноль).
- **R2** статистика грузится поздно — после рендера и после вёрстки формул.
- **R3** источник X/3 (`proto_last3_*`) не прогревается ни у кого: teacher-прогрев
  `warmTeacherModalStatsForStudent` (`:710`) греет только `questionStatsForTeacherV1`
  (`:749`), а `proto_last3` — нет; self-прогрева нет вовсе (`:711`).
- **R4** холодная страница = максимальная латентность (прокси `api.ege-trainer.ru`).

Опорные факты для решения:
- `buildProtoModalCards` (`:2692`) даёт карточки с `key` = `unic_id` (baseId) — то
  есть unic-ключи известны **сразу после манифеста, до рендера** → статистику можно
  загрузить до отрисовки бейджей.
- `refreshProtoModalBadges` зовётся в **двух** местах: `:2961` (открытие) и `:3301`
  (повторный refresh при смене студента) — рефактор обязан сохранить оба.
- Кеши на месте: `_SELF_PROTO_LAST3_CACHE` (`:626`), `_TEACHER_PROTO_LAST3_CACHE`
  (`:622`), `_TEACHER_MODAL_STATS_CACHE` (`:618`). Константы прогрева:
  `TEACHER_MODAL_PRELOAD_TTL_MS=10мин` (`:631`), `…CONCURRENCY=4` (`:632`).

## 3. Out of scope (НЕ делаем)

- ⛔️ **НЕ `git commit` / `git push`** — только после подтверждения оператора.
- НЕ трогаем SQL/RPC/провайдеры (`proto_last3_for_self_v1`, `*_for_teacher_*`,
  `homework.js`) — волна чисто отрисовочная, данные уже есть.
- НЕ трогаем **правильность** содержимого бейджа (это закрыто WMB1–5): меняем
  только **когда** и **в каком состоянии** он появляется.
- НЕ трогаем вёрстку формул (`typesetMathIfNeeded`) — реflow стема это отдельный
  визуал, не бейдж.
- НЕ трогаем аккордеонные бейджи раздела/подтемы (их разовый load-flash на старте
  страницы — отдельный, более слабый случай; плейсхолдер там нейтральный «—»).
- НЕ декомпозируем `picker.js` (W2). Правки аддитивные/локальные.
- НЕ вводим новый прогрев «на каждый рендер аккордеона» (сотни запросов) —
  прогрев только по раскрытию раздела (см. §5.5) и teacher-выбору студента.

## 4. Затрагиваемые файлы

Только `tasks/picker.js`:
- **Разделить** `refreshProtoModalBadges` на:
  - `loadProtoModalStatsMap(cards, opts)` — роле-зависимая ЗАГРУЗКА данных →
    `Map<unicKey, badgeStat>` (без DOM). self → `loadProtoLast3ForSelf`; teacher с
    выбранным студентом → `loadTeacherStatsForModal` + `loadProtoLast3ForModal`
    (+ `aggregateStatsForQuestionIds`); teacher без студента → пустая карта
    (состояние «Ученик не выбран»).
  - `applyProtoModalBadges(cards, statsMap)` — применение к DOM (текущие
    `setModalStatsBadge` + `setModalDateBadge`), с seq-guard.
- `openProtoPickerModal` (`:2900`): после манифеста — `const statsMap = await
  loadProtoModalStatsMap(cards)` (старт сразу, до вёрстки формул) → рендерить
  карточки **уже с бейджами** → `typesetMathIfNeeded` после. «Загрузка…» висит,
  пока грузятся манифест + статистика.
- `renderProtoModalCard`: принять опциональный `badgeStat` и строить бейдж сразу
  через `setModalStatsBadge(badge, badgeStat)` + `setModalDateBadge`. Если данных
  нет (teacher без студента / ошибка загрузки) — **нейтральный** статус («—» /
  «Ученик не выбран» / «…»), **никогда не «Не решал»** до подтверждения.
- Повторный путь (`:3301`): перевести на `applyProtoModalBadges(PROTO_MODAL_CARDS,
  await loadProtoModalStatsMap(...))` — поведение сохранить.
- **3b-прогрев teacher:** в `warmTeacherModalStatsForStudent` (`:710`) добавить
  прогрев `proto_last3` (вызвать `protoLast3ForTeacherV1` по unic пула → положить в
  `_TEACHER_PROTO_LAST3_CACHE`), рядом с существующим `questionStatsForTeacherV1`.
- **3b-прогрев self:** новый `warmSelfProtoLast3ForSection(section, opts)` — по
  раскрытию раздела грузит манифесты подтем раздела → unic ids →
  `protoLast3ForSelfV1` → `_SELF_PROTO_LAST3_CACHE`. Ограниченная конкуренция (как
  `TEACHER_MODAL_PRELOAD_CONCURRENCY`), TTL, отменяемость (seq), cancel при
  закрытии/смене.
- **Триггер прогрева:** в обработчике раскрытия раздела (`renderSectionNode`
  titleBtn click, `:2424`) — фоном звать self/teacher прогрев для подтем
  раскрытого раздела. Без блокировки UI.
- `node tools/bump_build.mjs`.

## 5. Пошаговый план

> **Task-tracking (обязательно):** в начале работы создай TaskList через
> `TaskCreate` с пунктами §5.1–§5.7, обновляй статус `TaskUpdate`
> (`in_progress`/`completed`) — чтобы оператор видел прогресс.

- **§5.1 — Разделить refresh на load+apply.** Вынести из `refreshProtoModalBadges`
  чистую загрузку `loadProtoModalStatsMap(cards, opts)` (роле-ветки → `Map<key,
  badgeStat>`) и `applyProtoModalBadges(cards, statsMap)` (DOM). Сам
  `refreshProtoModalBadges` оставить тонкой обёрткой `apply(cards, await load(...))`
  — чтобы повторный путь `:3301` не сломался.

- **§5.2 — Рендер-с-данными (3a).** В `openProtoPickerModal`: после `buildProtoModalCards`
  — `await loadProtoModalStatsMap(cards)` (старт до вёрстки формул); рендерить
  карточки, прокидывая `badgeStat` в `renderProtoModalCard`. «Загрузка…» снимается,
  только когда карточки готовы с бейджами. Сохранить `_PROTO_MODAL_SEQ`/seq-guard.

- **§5.3 — Нейтральный fallback вместо «Не решал».** В `renderProtoModalCard`
  бейдж без подтверждённых данных рисовать нейтрально («—»/«…»/«Ученик не выбран»),
  «Не решал» — только после подтверждённого нуля попыток (через `loadProtoModalStatsMap`).
  Это гарантия «никогда не дезинформируем», даже если 3a-путь где-то не сработал.

- **§5.4 — Прогрев teacher (3b).** В `warmTeacherModalStatsForStudent` добавить
  ветку прогрева `proto_last3` (`protoLast3ForTeacherV1` по unic пула →
  `_TEACHER_PROTO_LAST3_CACHE`), переиспользуя текущий worker-pool/TTL/seq.

- **§5.5 — Прогрев self (3b).** Новый `warmSelfProtoLast3ForSection` + вызов из
  обработчика раскрытия раздела (`:2424`). Грузит манифесты подтем раздела → unic →
  `protoLast3ForSelfV1` → `_SELF_PROTO_LAST3_CACHE`. Concurrency-лимит, TTL,
  отменяемость, cancel при закрытии модалки/смене раздела. Без блокировки клика.

- **§5.6 — Bump build.** `node tools/bump_build.mjs`, зафиксировать id в отчёте.

- **§5.7 — Governance + smoke.** Прогнать §9.

## 7. Риски и stop-ask точки

Не red-zone по `CURATOR.md §6.2`, но значимый рефактор потока shared-модуля
`picker.js` + новый фоновый прогрев. Scope lock: **только `tasks/picker.js`**.

Риски:
1. **Рефактор ломает открытие/повторный refresh.** Митигация: `refreshProtoModalBadges`
   остаётся тонкой обёрткой над load+apply; оба вызова (`:2961`, `:3301`)
   сохранены; seq-guard. Проверка — smoke обоих путей (открытие + смена студента).
2. **Прогрев = всплеск фоновых запросов / нагрузка на прокси.** Митигация:
   concurrency-лимит, TTL, прогрев **только по раскрытию раздела** (не на весь
   каталог), cancel при закрытии/смене. Stop-ask, если для приемлемого UX прогрев
   приходится вешать шире раздела.
3. **Холодный первый клик** всё равно платит один RPC (данные едут с сервера) —
   это честная «Загрузка…», не регрессия. Не считать дефектом.
4. **Cache-busting** — без `bump_build` фикс не доедет.

## 8. Критерии приёмки (DoD)

1. Открытие подтемы (оба роля) показывает бейджи прототипов **сразу корректными**
   (X/3 + дата при наличии попыток); промежуточного «Не решал»/«0/0» нет ни в
   одном пути. На холодном кеше — честная «Загрузка…», затем сразу верные карточки.
2. После раскрытия раздела (self) и выбора студента (teacher) открытие подтем
   этого раздела — **мгновенное из кеша**, без сетевого ожидания (3b работает).
3. Нейтральный fallback: при отсутствии подтверждённых данных бейдж не показывает
   «Не решал» (показывает «—»/«…»/«Ученик не выбран»).
4. Повторный refresh (смена студента у учителя) работает как раньше.
5. governance зелёные; charnet (student+teacher) зелёный (нерегрессия аккордеона).
6. `bump_build` выполнен, build id в отчёте.
7. **НЕ закоммичено / не запушено** — оставлено в рабочем дереве; отчёт это фиксирует.
