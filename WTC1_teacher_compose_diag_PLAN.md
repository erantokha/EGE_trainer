## 1. Цель

Пройти серию сценариев составления домашней работы на `home_teacher.html` (выбор ученика, добавление прототипов,
выбор всех 1–12, фильтры, переход в создание ДЗ / «Начать»), **воспроизвести и задокументировать** все некорректные

## 7. Риски и stop-ask точки

- **Побочные эффекты на живых данных:** не назначать ДЗ ученику, не слать уведомления, не удалять/менять данные ученика.
  Если сценарий объективно требует destructive-записи для воспроизведения → **STOP-ASK**.
- **Найден P0-баг порчи данных** (например, sync пишет чужому ученику) → задокументировать с доказательством, **НЕ чинить**,
  эскалировать немедленно (stop-ask 10c).
- **Соблазн «заодно починить»** очевидную мелочь → запрещено, это диагностика; фикс — отдельная волна.

## 6.3 Режим работы: автономный

> **Режим работы: автономный.** Не останавливайся за подтверждением на каждом сценарии; пройди все, собери отчёт.
> Куратор принимает целиком. Это **диагностика — НИЧЕГО не чинить**.
>
> **Останавливайся (stop-ask) только в экстренных случаях:**
> 1. Правка файла вне §4.
> 2. Заход в §3 (фикс продукта / назначение реальной ДЗ / destructive-запись) — даже «чтобы воспроизвести».
> 3. План противоречит реальности (элемент/RPC/механизм иной).
> 4. DoD недостижим без выхода за scope.
> 5. Governance/тест-инфра сломаны не по вине diag-спек.
> 6. Уязвимость/утечка креденшлов.
> 7. Задача распадается на независимые.
> 8. Diag-harness сам флейкает 2+ раза и мешает отличить баг продукта от шума теста.
> 9. Архитектурное решение вне §4.
> 10. **Проектные триггеры:**
>     - (a) сценарий требует назначить реальную ДЗ / послать уведомление, иначе не воспроизводится → STOP-ASK;
>     - (b) teacher-аккаунт/ученик не позволяет пройти сценарий (нет кандидатов/прав) → зафиксировать как «не покрыто», продолжить (НЕ стоп);
>     - (c) найден P0 (порча/утечка данных, кросс-ученик загрязнение) → задокументировать + STOP-ASK немедленно, НЕ чинить.
>
> **Что НЕ экстренный случай:** имена/порядок probes; решение, что сценарий «не воспроизводится детерминированно»
> (зафиксировать гипотезу из кода и идти дальше); повторные прогоны.
>
> **Формат stop-ask:** какой пункт, что обнаружено, доказательство, рекомендация.

## 10. Зачем именно так

Карта симптомов с доказательствами и корневыми гипотезами даёт узкую, дешёвую фикс-волну вместо «починим наугад».
Гибрид (авто-probes + инструментация + анализ кода) ловит и детерминированные баги (sync/дедуп/flush), и тайминг/сеть
(разлогин), которые в headless детерминированно не повторяются — для них честно фиксируем гипотезу из кода (как WHF2).

## 12. Приложение — сценарии (подтверждены оператором 2026-05-29)

Формат: **действие → что ловим (подозреваемая неправильная модель)**.

### T0 — приоритетные симптомы
- **T0.1 Спонтанный разлогин:** долго сидеть/собирать → внезапный logout, лечится refresh. Ловим: момент разрыва сессии
  (истёк токен без refresh? таймаут `getSession` на медленном VPS? транзиентный 401 без авто-refresh?), что показывает UI, почему refresh лечит.
- **T0.2 Не все задания добавились:** собрал работу → попали не все. Ловим: где теряются — resolve (вернулось меньше), дедуп (исключили лишнее), гонка дебаунса, или flush перед навигацией.

### A — сессия / auth
- **T-A1** idle дольше TTL токена → действие. Ловим: тихий refresh vs разлогин.
- **T-A2** действие при транзиентном сетевом сбое к VPS (offline 2–3с → online). Ловим: 401-ретрай/восстановление vs залипший logout.
- **T-A3** две вкладки teacher-home, в одной пере/раз-логин. Ловим: рассинхрон сессии.
- **T-A4** refresh в середине сборки. Ловим: восстановление сессии И added-set (персистентность контекста).

### B — выбор / переключение ученика
- **T-B1** выбрать ученика → статистика. Ловим: грузится init, combo не зависает, бейджи корректны.
- **T-B2** быстро переключать A→B→C. Ловим: race — не прилетает ли статистика прошлого ученика (seq).
- **T-B3** добавлять A, на середине переключиться на B. Ловим: утечка added-set A в контекст B.
- **T-B4** ученик с мало/ноль данных. Ловим: пустые состояния, подбор не падает.
- **T-B5** перевыбрать того же ученика. Ловим: дубли-перезагрузки, лишние RPC.

### C — добавление конкретных прототипов (прото-модалка)
- **T-C1** открыть модалку темы, +1 прототип. Ловим: счётчики обновились, задача в added-set.
- **T-C2** добавить N одного прототипа. Ловим: добавилось ровно N (а не меньше из-за shortage/дедупа), либо это видно.
- **T-C3** добавить → − убрать. Ловим: корректное снятие (trim), не залипает.
- **T-C4** прототипы из разных тем в одной сессии модалки. Ловим: все попали, per-card бейджи корректны.
- **T-C5** открыть → закрыть без изменений → открыть. Ловим: состояние не сбрасывается/не дублируется.

### D — выбор всех 1–12 / bulk
- **T-D1** «Выбрать всё» по 12 разделам. Ловим: сколько реально зарезолвилось vs запрошено (тихие per-section shortage).
- **T-D2** bulk pick all → сразу собрать/перейти (не дожидаясь). Ловим: успел ли движок дорезолвить до flush/навигации.
- **T-D3** «Сбросить всё» после bulk. Ловим: полная очистка + ротация seed, без остатков.
- **T-D4** bulk с активным фильтром. Ловим: фильтр учтён на всех 12, сколько отсеялось.

### E — корректность added-tasks (ядро «не все добавились»)
- **T-E1 shortage:** запросить больше, чем есть кандидатов (после exclude). Ловим: меньше возвращается — видно ли (shortage/warnings).
- **T-E2 дедуп через scope:** N по теме, затем по её разделу. Ловим: раздел исключает уже добавленные из темы — ожидаемо, но выглядит как «меньше».
- **T-E3 дебаунс-гонка:** быстро настучать +/+/+/−. Ловим: финальный added-set = финальному выбору (не промежуточному).
- **T-E4 уменьшение:** добавить много → понизить count. Ловим: trim убирает правильные/правильное число.
- **T-E5 пересборка контекста:** добавить → сменить фильтр → вернуть. Ловим: набор восстановился без потерь/дублей.
- **T-E6 global_all:** запросить весь объём. Ловим: число ≤ видимых разделов и совпадает с ожиданием.

### F — фильтры (unseen_low / stale / unstable)
- **T-F1** включить каждый фильтр → добавить. Ловим: подбор уважает фильтр; `filter_counts` ≈ реально добавленному.
- **T-F2** сменить фильтр с уже добавленными. Ловим: судьба прежнего added-set (per-(student+filter) контекст?).
- **T-F3** фильтр без кандидатов у ученика. Ловим: понятное пустое vs тихий ноль.

### G — переход в создание ДЗ
- **T-G1** собрать → «Создать ДЗ». Ловим: в prefill `hw_create` попали ровно все добавленные (counts + picked-refs) + ученик + фильтр. (НЕ submit'ить — стоп на prefill/preview.)
- **T-G2** добавить → немедленно «Создать ДЗ» (до оседания дебаунса). Ловим: `flushTeacherAddedTasksSelection` доводит sync — или часть теряется (сильный кандидат T0.2).
- **T-G3** создать → back на teacher-home. Ловим: восстановление выбора/added-set, нет дублей при повторной сборке.

### H — save-and-go / session-ссылка (учитель)
- **T-H1** собрать → «Начать» (trainer/list). Ловим: замороженный набор = добавленному; picked-refs/фильтр в ссылке.
- **T-H2** то же сразу после добавления (гонка flush, аналог T-G2).

### I — гонки / персистентность / edge
- **T-I1** refresh в середине активного resolve (in-flight RPC). Ловим: «полузастрявший» контекст; восстановление из sessionStorage.
- **T-I2** очень быстрые повторные клики «Создать ДЗ»/«Начать». Ловим: двойная отправка/навигация.
- **T-I3** долгая сессия сборки (много задач, разные scope). Ловим: деградация, рассинхрон счётчиков и реального added-set.

### Observability по каждому
RPC (`teacher_picking_screen_v2` init/resolve + `..._resolve_batch_v1`: `returned_n`/`shortage`/`warnings`/error),
число в added-модалке vs запрошенного, picked-refs в prefill vs добавленного, состояние сессии (token/exp) + 401, console-ошибки.
