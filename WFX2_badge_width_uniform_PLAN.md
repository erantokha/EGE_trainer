## 1. Цель

Сделать бейджи точности **одинаковой фиксированной ширины** (а не «по контенту»,
из-за чего у «0%» узко, у «100% 3/3» широко), на двух поверхностях и для обеих
ролей (ученик + учитель):
- **A.** В модалке подбора прототипов — бейдж точности выровнять по ширине с
  бейджем давности решения, все пилюли одной ширины.
- **B.** В аккордеоне подтем на главной — вернуть бейджам точности единую ширину.

## 2. Контекст и мотивация (по аудиту, эта сессия)

- **A (модалка `#protoPickerModal`):** бейдж точности `.proto-modal-badge` сейчас
  `min-width:0` (правка WFX-косметики 2026-06-05, `base.css:917`) → ширина по
  контенту → «прыгает» (узко у «0%»). Бейдж давности `.modal-date-badge` —
  `min-width:74px` (`base.css:919`). Стиль общий для обеих ролей (модалка — общий
  DOM, правил под роль нет). Цель: фиксированная единая ширина, точность = давность.
- **B (аккордеон `.home-topic-badge`):** ширина стала динамической из-за
  `width:auto`-override на десктопе — у учителя `base.css:2538`
  (`#accordion .badge:not(.home-section-pct):not(.home-section-cov)`), у ученика
  `pages/home-student.css:460` (зеркало, добавлено в WFX-косметике). До этого
  бейджи были одинаковой ширины через JS-вычисляемую переменную
  `--home-topic-badge-w` (`syncHomeTopicBadgesWidth` в `picker.js` измеряет самый
  широкий бейдж и задаёт всем одинаковую ширину; работает для обеих ролей через
  гейт `isStudentLikeHome`). Цель: вернуть единую ширину.

Подтверждения по коду:
- `tasks/trainer.html` — однозадачный раннер; `.stat-score` (`trainer.js:314,2204`)
  это итоговая сводка `#summary` («X/Y Z%»), не пер-тематический бейдж;
  `renderSmartPanel` (`trainer.js:1085`) рисует теги прогресса `done/target` без
  процентов. Пер-тематических бейджей точности там нет → Поверхность B = аккордеон.
- `--home-topic-badge-w` задаётся JS на `#accordion` (`picker.js:syncHomeTopicBadgesWidth`),
  правило `.home-topic-badge{width:var(--home-topic-badge-w, auto)}` есть у обеих
  ролей (`base.css:2579`, `home-student.css:449`) — но перебивается `width:auto`
  из `:not()`-правила (спецификация выше). Снятие override вернёт единую ширину.

### §5.6 — Доработка по фидбэку оператора (2026-06-05): подтемный бейдж = бейдж «%» раздела

После приёмки WFX2 оператор уточнил: бейджи подтем (`.home-topic-badge`) вышли
«суперузкими» (уравнивание `--home-topic-badge-w` дало узкую ширину по контенту «%»).
Нужно: подтемный бейдж точности — **той же фиксированной ширины, что бейдж «%»
раздела** (`.home-section-pct` = `--home-pct-w`, сейчас 40px), текст **по центру**.
Обе роли.

- `.home-topic-badge`: сменить ширину с `var(--home-topic-badge-w, auto)` на
  **`var(--home-pct-w)`**: `width:var(--home-pct-w) !important; min-width:var(--home-pct-w) !important`.
  - student: `home-student.css` (строка ~449, текущее `--home-topic-badge-w`);
  - teacher: `base.css` (строка ~2579, текущее `--home-topic-badge-w`).
- Текст по центру: на `.badge` уже есть `justify-content:center` — убедиться, что
  для топик-бейджа он не перебит; если контент визуально не по центру (из-за
  `margin-right:12px` / grid-placement) — добавить выравнивание (`justify-self`/
  `text-align:center`), чтобы «%» стоял по центру пилюли.
- `--home-topic-badge-w` (JS `syncHomeTopicBadgesWidth`) для ширины топик-бейджа
  больше не используется → JS НЕ трогаем (остаётся no-op для этого бейджа, вне scope).
- WFX2-правка `:not(.home-topic-badge)` в `:not()`-сбросе (§5.2/§5.3) **остаётся** —
  она и нужна, чтобы применилось правило `.home-topic-badge{width:…}`.
- Re-bump (свежий штамп, сверить с прод-штампом), governance, смок: скрин аккордеона
  обеих ролей — подтемные бейджи по ширине = бейджам «%» разделов, текст по центру.
- ⛔️ НЕ коммитить / НЕ пушить.

**DoD-уточнение:** п.2 DoD дополняется — подтемный бейдж точности равен по ширине
бейджу «%» раздела (`--home-pct-w`) и текст в нём по центру, на обеих ролях.

## 7. Риски и stop-ask точки

Не red-zone, но `base.css` — общий файл. Scope lock: только §4-файлы, только
ширина указанных бейджей.

Риски:
1. **Слишком узкая `<N>`** в §5.1 → обрезка/перенос «100% 3/3» или «Не решал».
   Митигация: подбор по смоку, проверить все состояния (включая «Загрузка…» и
   длинные значения). Брать с запасом.
2. **Снятие `width:auto` затронет не только топик-бейдж** (если `:not()` ловит
   другие не-section бейджи). Митигация: точечно исключить `.home-topic-badge`,
   проверить остальные бейджи аккордеона на смоке.
3. **`--home-topic-badge-w` не пересчитывается** в каком-то сценарии (раскрытие
   раздела, ресайз) → ширина не единая. Митигация: проверить смок при раскрытии
   нескольких разделов и ресайзе; `syncHomeTopicBadgesWidth` вызывается из
   рендера/ресайза — убедиться.
4. **Cache-busting** — без bump (со свежим штампом) правка не доедет.

## 8. Критерии приёмки (DoD)

1. **Модалка (обе роли):** все бейджи точности одной ширины; бейдж точности и бейдж
   давности — одинаковой ширины (выровнены). Контент не обрезан/не перенесён ни в
   одном состоянии (33%/0%/100% 3/3, «Не решал», «Загрузка…», дата).
2. **Аккордеон (обе роли):** бейджи точности подтем — одной ширины независимо от
   значения (0/33/67/100%); section-бейджи (%/покрытие) не изменились.
3. Никаких правок JS/SQL; затронуты только §4-файлы.
4. governance зелёные (`check_trainer_css_layers` и пр.); charnet student+teacher
   зелёные (нерегрессия аккордеона; модалка в снимок не входит).
5. `bump_build` со свежим штампом (отличным от прод-штампа), зафиксирован в отчёте.
6. **НЕ закоммичено / не запушено** — в рабочем дереве; отчёт это фиксирует.
