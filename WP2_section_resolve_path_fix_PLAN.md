## 1. Корень (доказан разведкой + reality-verify куратора)
Тесные клики «Выбрать всё» (≤300 мс, внутри debounce) коалесятся ДО старта синка → у всех 12 секций `delta=K≥2` → `canUseGlobalAll=false` (picker.js:3852, требует delta=1 у всех) → уходит в **section-batch** (`pickQuestionsViaTeacherScreenResolveBatch`, 12 section-requests, `complete:true`, picker.js:3873) → БД упирается в **statement_timeout → HTTP 500** (~8.8 с) → `batchRes` не Map → **фоллбэк: 12 ПОСЛЕДОВАТЕЛЬНЫХ `pickQuestionsViaTeacherScreenResolve` по секциям** (picker.js:3890-3901) ≈ 18 с. Итого ≈ 27 с.
**Мигание:** `#sum` мгновенно = желаемое (48), а предпросмотр (реальные `ctx.buckets`) ре-рендерится только в конце синка → 25-секундное расхождение sum↔предпросмотр, воспринимается как «скачут задачи».
Последовательно (зазор) — каждый синк видит delta=1 → дешёвый `global_all` (короткое замыкание, без 500) → ~4 с, норма.

## 2. Цель
- **(A)** Убрать обречённый 12-секционный батч для случая «все секции delta=K» — заменить на **K× `global_all`** (он быстрый, уважает `excludeQuestionIds`, в 500 не упирается). 28 с → ~3–8 с; больше нет 500 и серийного фоллбэка.
- **(C)** Убрать мигание: наполнять предпросмотр **инкрементально** (по мере добора) и/или показывать честный прогресс, чтобы `#sum` не расходился с тем, что реально готово.

## 5. Дизайн

**5.1. (A) K× global_all для uniform-секций.** В section-блоке (picker.js ~3850-3910): вычислить, все ли `remaining` секции = `SECTIONS.length` и имеют ОДИНАКОВУЮ delta = K (K≥1). Если да — **цикл K раундов**:
```js
for (let round = 0; round < K; round++) {
  if (seq !== _ADDED_SYNC_SEQ) return;
  const resolved = await pickQuestionsViaTeacherScreenResolve({
    request: { scope_kind: 'global_all', n: 1 },
    excludeTopicIds: Array.from(excludeTopics),
    excludeQuestionIds: getExcludeSet(),   // растёт по мере append → раунды дают РАЗНЫЕ задачи
  });
  if (seq !== _ADDED_SYNC_SEQ) return;
  for (const q of (Array.isArray(resolved) ? resolved : [])) {
    const secId = String(q?.section_id || '').trim();
    if (secId && sectionNeedMap.has(secId)) {
      appendPickedQuestionsToBucket(ctx, `section:${secId}`, [q]);  // инкрементально (см. 5.2/C)
    }
  }
  // (опц.) перерендер предпросмотра здесь → инкрементальный наполнитель
}
```
- Это покрывает K=1 (текущее поведение) и K≥2 (фикс). Существующую `canUseGlobalAll`-ветку (одиночный global_all) можно заменить этим унифицированным циклом.
- **Section-batch + 12-серийный фоллбэк (3873-3901) для uniform-случая больше НЕ вызывается** → нет 500, нет 18-с хвоста. (Оставить путь только для non-uniform, если решишь его не покрывать — §3.)

**5.2. (C) Инкрементальный предпросмотр + честный прогресс.** Цель — не показывать `#sum=48` при пустом предпросмотре 25 с.
- Наполнять предпросмотр **по мере добора** (перерендер после каждого раунда §5.1, или хотя бы после каждой завершённой секции) — задачи появляются прогрессивно, а не одним рывком в конце.
- (или/и) показывать прогресс «готово N / запрошено M» рядом с `#sum` или индикатор «подбираем…», пока синк идёт, чтобы число отражало РЕАЛЬНО готовое.
- После (A) окно рассинхрона и так схлопывается (28 с → ~6 с), так что (C) — полировка; делать минимально-достаточно, без переписывания рендера.

## 7. Риски и stop-ask
- **Эквивалентность подбора:** K× global_all даёт K задач на секцию (count/scope сохранены), но КОНКРЕТНЫЕ `question_id` могут отличаться от section-batch (even-distribution иначе раскладывает) — **допустимо**, зафиксировать. Уже добавленные сохраняются (append + растущий exclude). Если charnet пинит точные id — added-tasks вне stats-снимка (по прошлым волнам charnet их не покрывает) → ожидаемо зелёный; если поедет — проверить валидность и обосновать.
- **Растущий exclude между раундами обязателен** (иначе раунды дадут дубли) — `getExcludeSet()` после `appendPickedQuestionsToBucket` (он зовёт `incIdCount`). Проверить, что append реально пополняет exclude ДО следующего раунда.
- **seq-abort** (`if (seq !== _ADDED_SYNC_SEQ) return`) сохранить внутри цикла (как в существующих путях) — чтобы устаревший синк не дописывал.
- **Не сломать** flush/immediate/WP1-контроллер, proto/topic-добор, non-uniform путь.
- **Не уходить в SQL** (statement_timeout — это (B)). Любая правка вне picker.js/bump → STOP-ASK.

## 8. Критерии приёмки (DoD)
1. **(A)** «Выбрать всё» при тесном наложении (2 клика ≤150 мс) с учеником: **ноль HTTP 500** (`57014`), **ноль 12-серийного фоллбэка**; идёт K× `global_all`. Эмпирически: было ~27 с → стало **~3–8 с**.
2. Результат корректен: 24 задачи (2×12), count/scope/фильтр верны, добавленные сохраняются; конкретные id-сдвиги задокументированы.
3. **(C)** Нет мигания: предпросмотр наполняется инкрементально и/или прогресс честный; `#sum` не показывает финал при пустом предпросмотре длительно.
4. Последовательный сценарий (зазор) и proto/topic-добор — не сломаны; WP1-контроллер цел.
5. governance зелёный; `node --check picker.js`; bump_build; `git diff` — только picker.js (логика) + `?v=`.
6. **Замеры до/после** (тесное наложение): 500-статусы, число resolve и их scope (global_all vs section-batch), wall-time. В отчёте.

## 10. Зачем именно так
`global_all` — путь, который БД отрабатывает быстро (короткое замыкание), а 12-секционный `complete`-батч — нет (statement_timeout → 500 → 18-с серийный фоллбэк). Делая K× global_all, мы остаёмся на быстром пути для любого uniform-K, обходя серверное узкое место без правки SQL. Инкрементальный наполнитель убирает воспринимаемое мигание (расхождение желаемого и готового). Настоящий серверный фикс батча — (B), отдельно.
