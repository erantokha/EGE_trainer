## 1. Цель

Точно описать, **как устроен подбор задач** учителем (init→resolve, фильтры, scope), и **где именно** закодировано
поведение «фильтр = жёсткое условие, добора нет → подборка неполная». Локализовать:
(а) где фильтр применяется как hard-constraint; (б) почему даже БЕЗ фильтра resolve иногда возвращает < N;
(в) какие сигналы приоритета доступны в данных для fallback-добора. На выходе — **change-options** (BE / FE / hybrid)
с указанием точных мест (`file:line`), контрактного и red-zone-влияния, для реализации **«подборка всегда полная: задал N → получил N, фильтр как предпочтение с добором по приоритету»**.

## 2. Контекст и мотивация

Требование оператора: при сборке варианта фильтр/приоритет должен быть **предпочтением, а не отсевом** — если по
критерию (напр. «слабые») кандидатов не хватает, добивать следующим по приоритету, чтобы N всегда набиралось. Сейчас:
без фильтра часть задач не добавляется; с фильтром — тем более (нет «плохо решаемых» по теме → из темы ничего).

Grounding куратора (стартовая точка, проверить и углубить):
- Логика — **в SQL**: `teacher_picking_screen_v2.sql` (resolve) + `teacher_picking_resolve_batch_v1.sql` (батч-добор, основной путь). Фронт только запрашивает N и складывает возвращённое.
- Фильтр → флаги `is_weak/is_stale/is_unstable` (из `student_proto_state_v1`/`student_topic_state_v1`), CTE `candidate_base` (v2 ~стр.338, resolve_batch ~стр.338). `p_filter_id` валидируется на `unseen_low/stale/unstable`.
- Shortage — бэкенд-поведение: вернул меньше запрошенного + `is_shortage/returned_n` (ловится FE picker.js:3283, но это другой слой).

## 5. Пошаговый план

> **Task-tracking (обязательно):** TaskList через `TaskCreate` по §5.1–§5.6, статусы `TaskUpdate`.

**5.1. Трассировка BE resolve-пайплайна** (оба SQL). От входа (`p_scope_kind/scope_id/n/filter_id/exclude`) до выхода
(`picked_questions/shortage`): где и как (а) собираются кандидаты (`candidate_base`), (б) **применяется фильтр**
(hard `WHERE is_weak/...` или ранжирование?), (в) применяется `exclude_question_ids`/`exclude_topic_ids`,
(г) ранжирование/seed-порядок, (д) `LIMIT`/cap на N, (е) формируется `is_shortage/returned_n`. Точные `file:line` каждого звена.

**5.2. Два корня неполноты — раздельно:**
- **filtered short:** подтвердить, что фильтр — жёсткое условие (нет добора вне фильтра). Где именно отсекается.
- **no-filter short:** найти причину, почему БЕЗ фильтра resolve иногда < N (исчерпание банка после exclude? cap? scope-семантика global_all?). Это ключевой пункт — оператор жалуется и без фильтров.

**5.3. Инвентарь сигналов приоритета.** Что есть в `student_proto_state_v1`/`student_topic_state_v1` и в payload
(accuracy, is_weak/stale/unstable, last_attempt, seen-counts, reason `weak/low/stale/uncovered`), на чём можно
построить **fallback-лестницу** (без фильтра / при исчерпании фильтра). Зафиксировать доступные поля.

**5.4. FE-слой.** Как picker.js запрашивает N и обрабатывает returned<N (есть ли уже какой-либо добор/повтор?), и
**возможен ли FE-only fallback** (второй resolve-проход без фильтра на недостающее) без правки SQL — как дешёвая альтернатива BE-изменению.

**5.5. Change-options (≥3).** Сформулировать варианты реализации «complete-selection с priority-fallback»:
- **BE-A:** в SQL превратить фильтр в ранжирование + добор до N из остального пула (priority-ladder в `ORDER BY`/добор-CTE). Точные места, контрактное влияние (`picked_questions` те же поля? меняется ли `shortage`-семантика?), red-zone/миграция/smoke.
- **FE-B:** второй resolve-проход без фильтра (или по next-priority) на `requested_n − returned_n`, дедуп через exclude. Где в picker.js, плюсы/минусы (латентность, лишние RPC), без правки SQL.
- **Hybrid / иные** по факту находок.
Для каждого: что меняется, риск, контракт, нужен ли новый параметр RPC (напр. `p_fallback=true`), и **какое продуктовое решение требуется от оператора** (лестница приоритета; добор в рамках scope vs шире; пересекать ли границу фильтра).

**5.6. Отчёт** — §11.

## 7. Риски и stop-ask точки

- **Соблазн «сразу починить» SQL** — запрещено, read-only. Фикс — отдельная RED-ZONE волна с миграцией.
- Если для понимания нужно ВЫПОЛНИТЬ SQL против БД — не делать разрушительного; только чтение схемы/выборки при необходимости и с явной пометкой (или stop-ask).

## 6.3 Режим работы: автономный

> **Режим работы: автономный.** Доведи до DoD (карта + options), верни отчёт. Куратор принимает целиком. **Ничего не менять.**
>
> **Останавливайся (stop-ask) только в экстренных случаях:**
> 1. Любая попытка правки кода/SQL/миграции (вне read-only).
> 2. Заход за scope §3.
> 3. Реальность не совпадает с grounding §2 (логика не в этих SQL / фильтр устроен иначе) — зафиксировать и, если меняет всю картину, STOP-ASK.
> 4. Для разведки требуется выполнить не-read-only SQL против БД.
> 5. Уязвимость/утечка.
> 6. Найден баг данных/контракта, требующий немедленного решения.
> 7–9. (стандартные: распад задачи / повторный флейк / решение вне scope).
> 10. **Проектные триггеры:** (a) обнаружено, что «complete-selection» уже частично реализован где-то (флаг/ветка) — зафиксировать, не дублировать; (b) изменение требует данных, которых в state-вьюхах нет (приоритет невозможно построить) → зафиксировать как ограничение и STOP-ASK (это меняет осуществимость).
>
> **Формат stop-ask:** пункт, что обнаружено, влияние, рекомендация.

## 8. Критерии приёмки (DoD)

1. Полная карта resolve-пайплайна (BE v2 + resolve_batch) с `file:line` каждого звена §5.1.
2. Раздельно объяснены **filtered-short** и **no-filter-short** корни (почему N не набирается), с `file:line`.
3. Инвентарь доступных сигналов приоритета (поля state-вьюх/payload) — на чём строить fallback.
4. ≥3 change-options (BE-A / FE-B / hybrid) с точными местами, контрактным/red-zone влиянием, плюсами/минусами.
5. Явный список **продуктовых решений оператора** (лестница приоритета; scope добора; граница фильтра).
6. Оценка контракта: затрагиваются ли канонические RPC, нужен ли новый параметр/версия/миграция/smoke/governance.
7. Рекомендация куратору (какой вариант предпочтителен и почему) — это рекомендация, не решение.
8. `reports/wtc_resolve_recon_report.md` создан; код/SQL не изменены (`git diff` пуст по `docs/supabase`/`tasks`/`app`).

## 10. Зачем именно так

Точная карта «где hard-filter и почему N не набирается» + взвешенные options (BE-ранжирование vs FE-добор) дают
оператору осознанный выбор перед RED-ZONE правкой канонического контракта, а не «поменяем SQL наугад». Особенно важно
развести filtered-short и no-filter-short — у них может быть разный корень и разная стоимость фикса.
