# W13.0 — Контент-трек: конвертер банк → `content/` (часть 2, №13)

Первый исполняемый хэндофф трека «часть 2». **Офлайн, НЕ red-zone** — производит данные, рантайм/схему/
RLS не трогает. Контракт и решения — `docs/navigation/part2_integration_contract.md`; схема и спека
конвертера — `docs/navigation/part2_content_schema.md`.

## §1. Цель
Написать конвертер `~/Desktop/Вторая часть/13/_сборка/_export_trainer.py`, который из банка (75 клонов,
движок + солверы + `circle_svg`) генерирует **тела задач части 2 в формате `content/`** + **SVG-файлы
окружностей** + **теги класс/метод/приёмы**, ровно по `part2_content_schema.md`. Выход — готовые данные
для подключения в каталог (W13.1).

## §2. Контекст
Банк CAS-проверен (residual≈0). Эталоны (пошаговое решение + окружность) уже в нужном виде
(`_engine.solve_*` + `_card.circle_svg`). Стиль окружности утверждён оператором и запечён в `circle_svg`.
Тренажёр рендерит LaTeX через MathJax, фигуры через `<img>`.

## §3. Out of scope
- НЕ трогать каталог-метаданные / RPC / схему БД (это W13.1).
- НЕ менять движок банка (`_engine.py`, `_solver_*.py`, `_card.py`) кроме при необходимости вынести
  per-clone данные на уровень модуля для импорта (если ещё не вынесены) — без изменения логики.
- НЕ менять стиль окружности (визуал утверждён; перекраска — отдельно, через `circle_svg`).
- НЕ подключать задачи в рантайм/каталог, НЕ деплоить.

## §4. Затрагиваемые файлы
- Создать: `~/Desktop/Вторая часть/13/_сборка/_export_trainer.py`.
- Создать (выход): JSON-файлы части 2 + `…/img/part2/*.svg` в репозитории тренажёра (путь — согласовать
  с W13.1; до согласования писать в `reports/part2_content_draft/` репозитория, НЕ в `content/` напрямую).
- Только чтение: `_engine.py`, `_solver_*.py`, `_card.py`, `bank_13.md`, `_batch_*.json`.

## §5. Пошаговый план
> **Task-tracking (обязательно, `CURATOR.md §6.1`):** в начале создать TaskList через `TaskCreate` с
> пунктами §5.1–§5.6, обновлять `TaskUpdate` (`in_progress`/`completed`).

- **§5.1** Прочитать контракт и схему (`part2_integration_contract.md`, `part2_content_schema.md`).
  Свериться, что у каждого солвер-модуля per-clone данные доступны на уровне модуля (если нет — вынести,
  без изменения логики; самопроверки модулей должны остаться зелёными).
- **§5.2** Собрать единый список всех 75 клонов: `(cid, источник, класс, метод, приёмы, eqline_tex,
  параметры, солвер)`. Метод-полку взять по таблице раскладки из контракта (§«Классификация»).
- **§5.3** Для каждого клона вызвать солвер (интервал — `pick_interval_34`), собрать `stem`, `answer`,
  `solution.{steps,gen_groups,below}`, сгенерировать `circle_svg` и сохранить SVG-файл.
- **§5.4** Собрать JSON прототипа по `part2_content_schema.md` (поля `part/class/method/uses/max_primary/
  stem/answer_spec/answer/solution`). Сгруппировать по класс→метод→источник.
- **§5.5** Верификация (DoD): residual каждого `answer.roots` в исходный lhs < 1e-7; все 75 имеют
  `class/method/uses`; SVG-файлы созданы; JSON парсится. Напечатать сводку PASS/FAIL.
- **§5.6** Отчёт: путь к выходным файлам, сводка верификации, перечень класс→метод→источник→кол-во клонов.
