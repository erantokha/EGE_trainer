Дата обновления: 2026-03-31

Этот документ фиксирует продуктовую бизнес-логику teacher-facing фильтров для ручного подбора задач.

## 1. Purpose

Этот документ описывает:
- teacher manual picking на `home_teacher.html`;
- бизнес-логику teacher-фильтров;
- правила отбора задач для `proto / topic / section`;
- правила прозрачности UI для преподавателя.

Этот документ не описывает SQL-реализацию, RPC shape или frontend wiring. Он задаёт именно продуктовый источник правды.

## 2. Scope

Спецификация применяется к:
- ручному teacher-picking на `home_teacher.html`;
- выбору задач по `proto`;
- выбору задач по `topic`;
- выбору задач по `section`;
- глобальной кнопке `Выбрать все`.

Спецификация пока не применяется к:
- smart homework;
- комбинированию нескольких фильтров одновременно;
- будущему режиму `Работа на занятии`.

## 3. Core Principles

Общие принципы системы teacher-фильтров:
- в manual teacher-picking активен либо ровно один фильтр, либо ни одного;
- фильтры разделены по педагогическим целям;
- фильтр формирует строгий eligible pool;
- при нехватке задач система возвращает меньше задач, а не добирает их молча из других состояний;
- более узкий scope всегда сильнее более широкого.

## 4. Canonical Terms

Ниже фиксируется канонический словарь терминов.

### 4.1. `section`

Определение:
- верхнеуровневая тема каталога.

### 4.2. `topic`

Определение:
- подтема внутри `section`.

### 4.3. `proto`

Определение:
- уникальный прототип;
- в runtime соответствует `unic / type_id`.

### 4.4. `seen`

Определение:
- материал считается затронутым только по реальному `answer_event`.

Не считаются:
- открытие карточки;
- просмотр задания;
- выдача в ДЗ без попытки;
- любые frontend-only события.

### 4.5. `selection-session`

Черновое определение:
- текущая сессия teacher manual picking до явного `reset / rebuild / смены ученика`.

### 4.6. `reset`

Черновое определение:
- явный полный сброс текущего teacher selection-state.

### 4.7. `rebuild`

Черновое определение:
- явная пересборка выдачи по тем же критериям с новым random seed.

### 4.8. `eligible pool`

Определение:
- полный список `proto`, которые допускаются текущим фильтром и текущим scope.

## 5. Common Picking Rules

Этот раздел задаёт общие правила, одинаковые для всех teacher-фильтров.

### 5.1. Scope Priority

Иерархия scope:
- `proto > topic > section`

Следствия:
- явно выбранный `proto` резервируется за собой;
- явно выбранный `topic` резервируется за собой;
- `section` не имеет права повторно подбирать задачи из уже явно выбранного `topic`.

### 5.2. No Duplicate Questions

Общие правила:
- уже добавленные `question_id` не должны повторяться;
- уже выбранные в рамках текущего подбора `proto` не должны повторяться, если не оговорено иное в фильтре.

### 5.3. Picking Pipeline

Общий pipeline:
1. определяется eligible pool на уровне `proto`;
2. из eligible pool выбираются нужные `proto`;
3. для выбранных `proto` подбираются конкретные `question_id`.

### 5.4. Shortage Rule

Если eligible pool меньше запрошенного количества:
- система возвращает меньше задач;
- система не подмешивает задачи из других состояний;
- UI честно показывает недобор.

### 5.5. Global `Выбрать все`

Глобальная кнопка `Выбрать все` трактуется отдельно от локального изменения счётчиков:
- она работает по правилам текущего фильтра;
- её точное поведение фиксируется в разделе соответствующего фильтра.

## 6. Randomness Model

Этот раздел задаёт общую модель рандома для teacher manual picking.

### 6.1. General Rule

Выбор внутри приоритета случайный.

### 6.2. Seed-Based Behavior

Случайность должна быть:
- seed-based;
- стабильной в рамках текущей `selection-session`.

Следствия:
- одинаковые условия + одинаковый seed дают одинаковый результат;
- одинаковые условия + новый seed дают другой результат.

### 6.3. Seed Rotation

Seed должен меняться при:
- `Сбросить все`;
- явном `rebuild / regenerate`;
- смене ученика.

Seed не должен меняться от:
- открытия/закрытия модалок;
- просмотра уже добавленных задач;
- незначительных UI-событий, не означающих явную пересборку.

## 7. Filter 1: `Не решал / мало решал`

### 7.1. Pedagogical Goal

Фильтр `Не решал / мало решал` — это строгий диагностический фильтр.

Он используется для:
- первичной диагностики нового ученика;
- добора статистики по ещё не обследованным прототипам;
- расширения покрытия по теме без смешивания с логикой слабости, точности или закрепления.

Фильтр не отвечает на вопросы:
- где ученик слабый;
- где у него низкая точность;
- где раньше получалось или не получалось;
- где нужен повтор ранее освоенного материала.

Эти сценарии относятся к другим фильтрам.

### 7.2. Data Source

Все статусы и eligibility для этого фильтра считаются только по реальным:
- `answer_event`

Не учитываются:
- открытие карточки;
- просмотр задания;
- выдача задания в ДЗ без попытки;
- любые frontend-only события.

### 7.3. Status Definitions

Статусы задаются отдельно на уровне `topic` и `proto`.

Статусы `topic`:
- `not_seen`
  Критерий: в подтеме затронуто `0` уникальных прототипов.
- `low_seen`
  Критерий: в подтеме затронуто строго меньше `3` уникальных прототипов, но не `0`.
- `enough_seen`
  Критерий: в подтеме затронуто `3` или более уникальных прототипов.

Статусы `proto`:
- `not_seen`
  Критерий: `unique_question_ids_seen = 0`.
- `low_seen`
  Критерий: `unique_question_ids_seen < 2`, но не `0`.
- `enough_seen`
  Критерий: `unique_question_ids_seen >= 2`.

Для этого фильтра именно `topic` и `proto` являются базовыми сущностями диагностики. Статусы должны быть взаимоисключающими.

### 7.4. Eligibility Rules

Главная единица отбора:
- `уникальный прототип` (`proto / type_id`)

Это означает:
1. Система сначала определяет eligible pool на уровне `proto`.
2. Затем выбирает нужные `proto` по каскаду приоритетов.
3. Только после этого подбирает конкретные `question_id` внутри выбранных `proto`.

Для этого фильтра eligible считаются:
- `proto.not_seen`
- `proto.low_seen`

Не eligible:
- `proto.enough_seen`

Внутри уже выбранного `proto` система должна по возможности сначала выбирать ещё не затронутые `question_id` этого `proto`.

Если этого недостаточно, допускается добор уже затронутыми `question_id`, но:
- без повторов уже добавленных задач;
- без повторов внутри текущего набора.

### 7.5. Priority Cascade

Для широкого scope фильтр использует каскад из четырёх приоритетов:

1. Уникальные прототипы из подтем со статусом `topic.not_seen`.
2. Все оставшиеся уникальные прототипы по всей теме со статусом `proto.not_seen`.
3. Уникальные прототипы из подтем со статусом `topic.low_seen`.
4. Все оставшиеся уникальные прототипы по всей теме со статусом `proto.low_seen`.

Правило перехода между приоритетами:
- сначала берётся всё возможное из приоритета `1`;
- если не хватило, система переходит к приоритету `2`;
- если всё ещё не хватило, система переходит к приоритету `3`;
- затем к приоритету `4`.

Внутри каждого приоритета:
- собирается один общий пул кандидатов;
- из него выбираются нужные `proto` случайно;
- уже выбранные `proto` повторно не участвуют.

### 7.6. Behavior In `proto`

Если teacher явно выбрал конкретный `proto`, фильтр работает бинарно:

- если выбранный `proto` имеет статус `not_seen` или `low_seen`, он eligible;
- если выбранный `proto` имеет статус `enough_seen`, он не eligible.

Если `proto` eligible:
- система подбирает до `N` задач внутри этого `proto`;
- сначала по возможности используются ещё не затронутые `question_id`;
- если их не хватает, допускается добор уже затронутыми `question_id`.

Если `proto` не проходит фильтр:
- система возвращает `0` задач по этому запросу;
- UI должен показать понятное сообщение, что выбранный `proto` не попадает в фильтр `Не решал / мало решал`.

### 7.7. Behavior In `topic`

У `topic` нет внутренних подтем, поэтому каскад сокращается:

1. Все `proto.not_seen` внутри выбранной `topic`.
2. Если не хватило, все `proto.low_seen` внутри выбранной `topic`.

Дальше:
- из кандидатов по каждому priority-layer случайно выбираются нужные `proto`;
- затем внутри выбранных `proto` подбираются конкретные `question_id`;
- если кандидатов меньше, чем запрошено, система возвращает меньше задач.

Для `topic` не существует отдельного шага "через вложенные подтемы", так как ниже `topic` уже находятся только `proto`.

### 7.8. Behavior In `section`

Для `section` используется полный четырёхступенчатый каскад, описанный в разделе `7.5`.

Дополнительное правило для `section`:
- если teacher уже явно выбрал один или несколько `topic` внутри этой `section`, то `section` не имеет права повторно добирать задачи из этих `topic`;
- `section` должна подбирать только из оставшихся `topic`.

Это правило обеспечивает приоритет:
- `proto > topic > section`

Если teacher выбрал `section` и запросил `N` задач:
- система идёт по каскаду приоритетов `1 -> 2 -> 3 -> 4`;
- на каждом шаге учитывает только ещё не исключённые `topic` и ещё не выбранные `proto`;
- после выбора `proto` подбирает конкретные `question_id`.

### 7.9. Global `Выбрать все`

Глобальная кнопка `Выбрать все` при активном фильтре `Не решал / мало решал` работает так:

- система проходит по каждой верхнеуровневой теме (`section`);
- для каждой темы проверяет, существует ли хотя бы один eligible candidate по этому фильтру;
- если в теме есть хотя бы один candidate, система добавляет ровно `1` задачу из этой темы;
- подбор внутри каждой темы всё равно идёт по тому же priority cascade, что и для обычного section-pick.

Это означает:
- глобальная кнопка не выбирает все подходящие задачи;
- глобальная кнопка не выбирает все подходящие `proto`;
- глобальная кнопка даёт по одной диагностической задаче из каждой темы, где это возможно.

### 7.10. Shortage Behavior

Фильтр работает строго.

Если по выбранному scope и по всем приоритетам удалось подобрать меньше задач, чем запросил teacher:
- система возвращает только найденное количество;
- система не добирает задачи из других состояний;
- система не снимает фильтр автоматически;
- UI честно показывает недобор.

Пример текста:
- `Подобрано 2 из 11 по фильтру "Не решал / мало решал"`

### 7.11. Examples

Пример 1: `section`

Teacher запросил `5` задач из темы `1. Планиметрия`.

Система:
1. ищет `proto` из `topic.not_seen`;
2. если после этого задач меньше `5`, добирает `proto.not_seen` по всей теме;
3. если всё ещё меньше `5`, добирает `proto` из `topic.low_seen`;
4. затем добирает `proto.low_seen` по всей теме.

Если после всех шагов найдено только `2`, система возвращает `2`.

Пример 2: `topic`

Teacher запросил `3` задачи из `1.1`.

Система:
1. берёт все `proto.not_seen` внутри `1.1`;
2. если этого мало, добирает `proto.low_seen` внутри `1.1`;
3. если суммарно найдено только `1`, система возвращает `1`.

Пример 3: `section` + уже выбранный `topic`

Teacher сначала выбрал `topic 1.1`, затем выбрал `section 1`.

Система:
- не имеет права повторно брать задачи из `1.1`;
- подбирает задачи для `section 1` только из остальных подтем этой темы;
- при этом всё равно использует каскад `not_seen -> low_seen`.

## 8. Filter 2: `Давно решал`

### 8.1. Pedagogical Goal

Фильтр `Давно решал` — это строгий фильтр на повторение ранее освоенного материала.

Он используется для:
- повторения уже знакомых прототипов;
- возвращения к материалу, который раньше уже получался у ученика;
- поддержания навыка без смешивания с диагностикой нового материала и без смешивания с логикой текущей нестабильности.

Фильтр не отвечает на вопросы:
- где у ученика мало статистики;
- где материал ещё не обследован;
- где у ученика низкая точность;
- где у ученика есть текущая нестабильность.

Эти сценарии относятся к другим фильтрам.

### 8.2. Data Source

Все статусы и eligibility для этого фильтра считаются только по реальным:
- `answer_event`

Не учитываются:
- открытие карточки;
- просмотр задания;
- выдача задания в ДЗ без попытки;
- любые frontend-only события.

### 8.3. Status Definitions

Для этого фильтра используется канонический статус `stale`, согласованный с архитектурным 4-layer контрактом.

Статусы `proto`:
- `stale`
  Критерий: одновременно выполнены условия:
  - `has_independent_correct = true`;
  - `weak = false`;
  - накоплено не менее `2` реальных попыток;
  - последний `answer_event` старше `30` дней.
- `fresh_or_ineligible`
  Любой `proto`, не попадающий в `stale`.

Таким образом:
- `proto` считается `stale` только если это уже ранее освоенный и не выглядящий слабым материал, который давно не повторялся;
- `proto` не считается `stale`, если он просто давно не открывался, но никогда не был реально освоен.

Статусы `topic`:
- `stale`
  Критерий: внутри `topic` есть ранее освоенный материал, и последний `answer_event` по уже освоенным `proto` этой `topic` старше `30` дней.
- `fresh_or_mixed`
  Любая `topic`, не попадающая в `stale`.

Смысл:
- `topic.stale` — это подтема, к которой ученик как к уже знакомому материалу давно не возвращался;
- `topic.fresh_or_mixed` — это либо свежая подтема, либо подтема со смешанной картиной, где есть и старый, и относительно свежий материал.

### 8.4. Eligibility Rules

Главная единица отбора:
- `уникальный прототип` (`proto / type_id`)

Это означает:
1. Система сначала определяет eligible pool на уровне `proto`.
2. Затем выбирает нужные `proto` по правилам фильтра.
3. Только после этого подбирает конкретные `question_id` внутри выбранных `proto`.

Для этого фильтра eligible считаются:
- только `proto.stale`

Не eligible:
- `proto`, который никогда не был решён самостоятельно;
- `proto`, который сейчас выглядит `weak`;
- `proto`, который практиковался недавно;
- `proto`, по которому накоплено слишком мало попыток, чтобы считать его реально пройденным.

Внутри выбранного `proto` система по возможности сначала выбирает ещё не затронутые аналоги этого `proto`, чтобы teacher повторял сам тип задачи, а не обязательно тот же самый экземпляр.

Если ещё не затронутых аналогов недостаточно, допускается добор уже ранее встречавшимися `question_id`, но:
- без повторов уже добавленных задач;
- без повторов внутри текущего набора.

### 8.5. Priority Cascade

Для широкого scope фильтр использует двухступенчатый каскад:

1. Уникальные прототипы из подтем со статусом `topic.stale`.
2. Все оставшиеся уникальные прототипы по всей теме со статусом `proto.stale`.

Правило перехода:
- сначала берётся всё возможное из приоритета `1`;
- если не хватило, система переходит к приоритету `2`.

Дополнительное правило внутри одного priority-layer:
- более давно не повторявшийся материал имеет более высокий внутренний приоритет;
- для этого используются age buckets:
  - `90+` дней;
  - `60-89` дней;
  - `30-59` дней;
- внутри одного bucket выбор происходит случайно, но seed-based.

Смысл:
- сначала teacher получает материал из действительно "залежавшихся" подтем;
- если этого недостаточно, система добирает отдельные `stale`-прототипы из более смешанной картины по теме.

### 8.6. Behavior In `proto`

Если teacher явно выбрал конкретный `proto`, фильтр работает бинарно:

- если выбранный `proto` имеет статус `stale`, он eligible;
- если выбранный `proto` не имеет статус `stale`, он не eligible.

Если `proto` eligible:
- система подбирает до `N` задач внутри этого `proto`;
- по возможности сначала берёт ещё не затронутые аналоги;
- затем при необходимости добирает уже ранее встречавшимися аналогами.

Если `proto` не проходит фильтр:
- система возвращает `0` задач по этому запросу;
- UI должен показать понятное сообщение, что выбранный `proto` не попадает в фильтр `Давно решал`.

### 8.7. Behavior In `topic`

Если teacher запрашивает `N` задач из `topic`, система работает только внутри этой `topic`.

Алгоритм:
1. Собираются все `proto.stale` внутри выбранной `topic`.
2. Кандидаты раскладываются по age buckets:
   - `90+` дней;
   - `60-89` дней;
   - `30-59` дней.
3. Сначала используются кандидаты из более старших bucket.
4. Внутри одного bucket используется seed-based random.
5. Из выбранных `proto` подбираются конкретные `question_id`.

Если `stale`-прототипов меньше, чем запрошено:
- система возвращает меньше задач;
- системой не подмешиваются другие `proto`, которые не проходят фильтр.

### 8.8. Behavior In `section`

Если teacher запрашивает `N` задач из `section`, система использует полный двухступенчатый каскад, описанный в разделе `8.5`.

Дополнительное правило для `section`:
- если teacher уже явно выбрал один или несколько `topic` внутри этой `section`, то `section` не имеет права повторно добирать задачи из этих `topic`;
- `section` должна подбирать только из оставшихся `topic`.

Это правило обеспечивает приоритет:
- `proto > topic > section`

Если teacher выбрал `section` и запросил `N` задач:
- система сначала ищет `proto` из `topic.stale`;
- если этого недостаточно, добирает оставшиеся `proto.stale` по всей теме;
- внутри каждого priority-layer сначала использует более старшие age buckets;
- внутри одного bucket использует seed-based random;
- после выбора `proto` подбирает конкретные `question_id`.

### 8.9. Global `Выбрать все`

Глобальная кнопка `Выбрать все` при активном фильтре `Давно решал` работает так:

- система проходит по каждой верхнеуровневой теме (`section`);
- для каждой темы проверяет, существует ли хотя бы один eligible candidate по фильтру;
- если в теме есть хотя бы один candidate, система добавляет ровно `1` задачу из этой темы;
- подбор внутри каждой темы идёт по тем же правилам, что и для обычного section-pick:
  - сначала `topic.stale`,
  - затем оставшиеся `proto.stale`,
  - с приоритетом более старших age buckets.

Это означает:
- глобальная кнопка не выбирает все `stale`-задачи;
- глобальная кнопка не выбирает все `stale`-прототипы;
- глобальная кнопка даёт по одной задаче на повторение из каждой темы, где это возможно.

### 8.10. Shortage Behavior

Фильтр работает строго.

Если по выбранному scope и по всем приоритетам удалось подобрать меньше задач, чем запросил teacher:
- система возвращает только найденное количество;
- система не добирает задачи из других состояний;
- система не снимает фильтр автоматически;
- UI честно показывает недобор и его причину.

Пример текста:
- `Подобрано 3 из 8: найдено только 3 прототипа со статусом "Давно решал"`

### 8.11. Examples

Пример 1: `section`

Teacher запросил `4` задачи из темы `1. Планиметрия`.

Система:
1. ищет `proto` из `topic.stale`;
2. внутри найденных кандидатов сначала рассматривает bucket `90+`, затем `60-89`, затем `30-59`;
3. при равенстве внутри bucket использует seed-based random;
4. если после этого задач меньше `4`, добирает остальные `proto.stale` по всей теме по тем же правилам.

Если после всех шагов найдено только `2`, система возвращает `2`.

Пример 2: `topic`

Teacher запросил `2` задачи из `1.1`.

Система:
1. берёт только `proto.stale` внутри `1.1`;
2. раскладывает их по age buckets;
3. сначала использует более старшие bucket;
4. при равенстве внутри bucket использует seed-based random;
5. если найден только `1` подходящий `proto`, возвращает `1`.

Пример 3: `section` + уже выбранный `topic`

Teacher сначала выбрал `topic 1.1`, затем выбрал `section 1`.

Система:
- не имеет права повторно брать задачи из `1.1`;
- подбирает задачи для `section 1` только из остальных подтем этой темы;
- при этом всё равно использует фильтр `Давно решал`, его двухступенчатый каскад и age buckets.

## 9. Filter 3: `Нестабильно решает`

### 9.1. Pedagogical Goal

Фильтр `Нестабильно решает` — это строгий фильтр на закрепление ранее освоенного, но сейчас неустойчивого материала.

Он используется для:
- закрепления прототипов, которые у ученика раньше уже получались;
- доработки материала, по которому сейчас видно нестабильное выполнение;
- перевода навыка из состояния "иногда получается" в состояние устойчивого владения.

Фильтр не отвечает на вопросы:
- где у ученика мало статистики;
- где материал ещё не обследован;
- где материал давно не повторялся, но сейчас не выглядит слабым;
- где у ученика вообще не было самостоятельного успешного решения.

Эти сценарии относятся к другим фильтрам.

### 9.2. Data Source

Все статусы и eligibility для этого фильтра считаются только по реальным:
- `answer_event`

Не учитываются:
- открытие карточки;
- просмотр задания;
- выдача задания в ДЗ без попытки;
- любые frontend-only события.

### 9.3. Status Definitions

Для этого фильтра используется канонический статус `weak`, согласованный с архитектурным 4-layer контрактом.

Статусы `proto`:
- `unstable`
  Критерий: одновременно выполнены условия:
  - `has_independent_correct = true`;
  - `weak = true`.
- `stable_or_ineligible`
  Любой `proto`, не попадающий в `unstable`.

Таким образом:
- `proto` считается `unstable` только если это материал, который раньше уже хотя бы раз получался самостоятельно;
- `proto` не считается `unstable`, если по нему просто `0%` без единого самостоятельного успеха;
- `proto` не считается `unstable`, если он давно не повторялся, но сейчас не выглядит `weak`.

Статусы `topic`:
- `unstable`
  Критерий: topic-level rollup по ранее освоенным `proto` внутри этой `topic` имеет статус `weak`.
- `stable_or_mixed`
  Любая `topic`, не попадающая в `unstable`.

Смысл:
- `topic.unstable` — это подтема, в которой слабость видна уже на уровне общего rollup, а не только по одному случайному `proto`;
- `topic.stable_or_mixed` — это подтема без выраженного общего сигнала на закрепление или со смешанной картиной.

### 9.4. Eligibility Rules

Главная единица отбора:
- `уникальный прототип` (`proto / type_id`)

Это означает:
1. Система сначала определяет eligible pool на уровне `proto`.
2. Затем выбирает нужные `proto` по правилам фильтра.
3. Только после этого подбирает конкретные `question_id` внутри выбранных `proto`.

Для этого фильтра eligible считаются:
- только `proto.unstable`

Не eligible:
- `proto`, который никогда не был решён самостоятельно;
- `proto`, по которому мало попыток и недостаточно данных;
- `proto`, который сейчас не имеет статуса `weak`;
- `proto`, который попадает под `stale`, а не под `unstable`.

`stale` и `unstable` должны быть взаимоисключающими состояниями:
- `stale` требует `weak = false`;
- `unstable` требует `weak = true`.

Внутри выбранного `proto` система по возможности сначала выбирает ещё не затронутые аналоги этого `proto`, чтобы teacher закреплял сам тип задачи, а не только память о конкретном экземпляре.

Если ещё не затронутых аналогов недостаточно, допускается добор уже ранее встречавшимися `question_id`, но:
- без повторов уже добавленных задач;
- без повторов внутри текущего набора.

### 9.5. Priority Cascade

Для широкого scope фильтр использует двухступенчатый каскад:

1. Уникальные прототипы из подтем со статусом `topic.unstable`.
2. Все оставшиеся уникальные прототипы по всей теме со статусом `proto.unstable`.

Правило перехода:
- сначала берётся всё возможное из приоритета `1`;
- если не хватило, система переходит к приоритету `2`.

Дополнительное правило внутри одного priority-layer:
- чем ниже accuracy, тем выше внутренний приоритет;
- при одинаковой accuracy выше приоритет у более свежей проблемы:
  - более недавний `last_attempt_at` идёт выше;
- при одинаковой accuracy и одинаковой свежести выше приоритет у `proto` с большим числом попыток;
- при полном равенстве используется seed-based random.

Смысл:
- сначала teacher получает материал из явно проблемных подтем;
- если этого недостаточно, система добирает отдельные unstable-прототипы из более смешанной картины по теме;
- свежая нестабильность важнее старого слабого хвоста.

### 9.6. Behavior In `proto`

Если teacher явно выбрал конкретный `proto`, фильтр работает бинарно:

- если выбранный `proto` имеет статус `unstable`, он eligible;
- если выбранный `proto` не имеет статус `unstable`, он не eligible.

Если `proto` eligible:
- система подбирает до `N` задач внутри этого `proto`;
- по возможности сначала берёт ещё не затронутые аналоги;
- затем при необходимости добирает уже ранее встречавшимися аналогами.

Если `proto` не проходит фильтр:
- система возвращает `0` задач по этому запросу;
- UI должен показать понятное сообщение, что выбранный `proto` не попадает в фильтр `Нестабильно решает`.

### 9.7. Behavior In `topic`

Если teacher запрашивает `N` задач из `topic`, система работает только внутри этой `topic`.

Алгоритм:
1. Собираются все `proto.unstable` внутри выбранной `topic`.
2. Кандидаты ранжируются по accuracy:
   - чем ниже accuracy, тем выше приоритет.
3. При одинаковой accuracy выше ставятся более свежие проблемы:
   - более недавний `last_attempt_at` идёт выше.
4. При одинаковой accuracy и одинаковой свежести выше ставятся `proto` с большим количеством попыток.
5. При полном равенстве используется seed-based random.
6. Из выбранных `proto` подбираются конкретные `question_id`.

Если `unstable`-прототипов меньше, чем запрошено:
- система возвращает меньше задач;
- системой не подмешиваются другие `proto`, которые не проходят фильтр.

### 9.8. Behavior In `section`

Если teacher запрашивает `N` задач из `section`, система использует полный двухступенчатый каскад, описанный в разделе `9.5`.

Дополнительное правило для `section`:
- если teacher уже явно выбрал один или несколько `topic` внутри этой `section`, то `section` не имеет права повторно добирать задачи из этих `topic`;
- `section` должна подбирать только из оставшихся `topic`.

Это правило обеспечивает приоритет:
- `proto > topic > section`

Если teacher выбрал `section` и запросил `N` задач:
- система сначала ищет `proto` из `topic.unstable`;
- если этого недостаточно, добирает оставшиеся `proto.unstable` по всей теме;
- внутри каждого priority-layer сначала использует кандидатов с худшей accuracy;
- при равной accuracy использует более свежую проблему как tie-break;
- затем использует большее число попыток;
- при полном равенстве использует seed-based random;
- после выбора `proto` подбирает конкретные `question_id`.

### 9.9. Global `Выбрать все`

Глобальная кнопка `Выбрать все` при активном фильтре `Нестабильно решает` работает так:

- система проходит по каждой верхнеуровневой теме (`section`);
- для каждой темы проверяет, существует ли хотя бы один eligible candidate по фильтру;
- если в теме есть хотя бы один candidate, система добавляет ровно `1` задачу из этой темы;
- подбор внутри каждой темы идёт по тем же правилам, что и для обычного section-pick:
  - сначала `topic.unstable`,
  - затем оставшиеся `proto.unstable`,
  - с приоритетом у худшей accuracy.

Это означает:
- глобальная кнопка не выбирает все unstable-задачи;
- глобальная кнопка не выбирает все unstable-прототипы;
- глобальная кнопка даёт по одной задаче на закрепление из каждой темы, где это возможно.

### 9.10. Shortage Behavior

Фильтр работает строго.

Если по выбранному scope и по всем приоритетам удалось подобрать меньше задач, чем запросил teacher:
- система возвращает только найденное количество;
- система не добирает задачи из других состояний;
- система не снимает фильтр автоматически;
- UI честно показывает недобор и его причину.

Пример текста:
- `Подобрано 2 из 6: найдено только 2 прототипа со статусом "Нестабильно решает"`

### 9.11. Examples

Пример 1: `section`

Teacher запросил `4` задачи из темы `3. Стереометрия`.

Система:
1. ищет `proto` из `topic.unstable`;
2. внутри найденных кандидатов выше ставит более низкую accuracy;
3. при равной accuracy выше ставит более свежую проблему;
4. при равной accuracy и свежести выше ставит `proto` с большим числом попыток;
5. если после этого задач меньше `4`, добирает остальные `proto.unstable` по всей теме по тем же правилам.

Если после всех шагов найдено только `2`, система возвращает `2`.

Пример 2: `topic`

Teacher запросил `2` задачи из `5.1`.

Система:
1. берёт только `proto.unstable` внутри `5.1`;
2. сортирует их по accuracy от худшей к лучшей;
3. при равной accuracy учитывает свежесть проблемы;
4. затем учитывает число попыток;
5. если найден только `1` подходящий `proto`, возвращает `1`.

Пример 3: `section` + уже выбранный `topic`

Teacher сначала выбрал `topic 7.2`, затем выбрал `section 7`.

Система:
- не имеет права повторно брать задачи из `7.2`;
- подбирает задачи для `section 7` только из остальных подтем этой темы;
- при этом всё равно использует фильтр `Нестабильно решает` и его двухступенчатый каскад.

| `Не решал / мало решал` | Если по всем приоритетам нашлось меньше задач, чем запрошено, система возвращает только найденное количество и честно показывает недобор. |
| `Давно решал` | Если по всем приоритетам нашлось меньше задач, чем запрошено, система возвращает только найденное количество и честно показывает недобор. |
| `Нестабильно решает` | Если по всем приоритетам нашлось меньше задач, чем запрошено, система возвращает только найденное количество и честно показывает недобор. |

Общее правило:
- система не подмешивает задачи из других фильтровых состояний;
- система не снимает фильтр автоматически;
- shortage трактуется как нормальное и прозрачное поведение, а не как скрытый fallback.

### 10.7. `reset / rebuild`

| Filter | Canonical Behavior |
| --- | --- |
| `Не решал / мало решал` | При одинаковых условиях и том же `seed` выдача стабильна; после явного `reset / rebuild / смены ученика` генерируется новый `seed`, и teacher может получить другой набор по тем же критериям. |
| `Давно решал` | При одинаковых условиях и том же `seed` выдача стабильна; после явного `reset / rebuild / смены ученика` генерируется новый `seed`, и teacher может получить другой набор по тем же критериям. |
| `Нестабильно решает` | При одинаковых условиях и том же `seed` выдача стабильна; после явного `reset / rebuild / смены ученика` генерируется новый `seed`, и teacher может получить другой набор по тем же критериям. |

Общее правило:
- без явной пересборки выдача не должна "плавать" сама по себе от второстепенных UI-действий.

## 11. UI Transparency Rules

Здесь фиксируются обязательные UI-правила прозрачности для teacher:
- сообщение о недоборе;
- сообщение, если явный `proto` не проходит фильтр;
- объяснение, что после `reset / rebuild` выдача может измениться.

### 11.1. Active Filter Visibility

UI должен явно показывать teacher, какой фильтр сейчас активен.

Обязательные требования:
- в manual picking одновременно может быть активен только один teacher-фильтр;
- активное состояние фильтра должно быть визуально однозначным;
- teacher не должен гадать, по какому именно правилу сейчас идёт подбор.

Если фильтр не выбран:
- UI должен трактовать это как нейтральный режим без teacher-фильтра.

### 11.2. Student Requirement

Teacher-фильтры работают только при выбранном ученике.

Обязательные требования:
- если ученик не выбран, teacher-фильтры должны быть disabled или визуально очевидно неактивны;
- UI должен явно объяснять это ограничение.

Пример текста:
- `Фильтры работают только при выбранном ученике`

### 11.3. Shortage Messaging

Если по активному фильтру удалось подобрать меньше задач, чем запросил teacher, UI обязан показывать недобор явно.

Обязательные требования:
- UI должен показывать фактическое количество найденных задач;
- UI должен показывать, что недобор связан именно с фильтром, а не со случайной ошибкой;
- UI не должен маскировать shortage молчаливой добивкой задачами из других состояний.

Минимальный формат сообщения:
- `Подобрано X из Y`

Предпочтительный формат сообщения:
- `Подобрано X из Y: найдено только X прототипов по фильтру "<название фильтра>"`

Примеры:
- `Подобрано 2 из 11 по фильтру "Не решал / мало решал"`
- `Подобрано 3 из 8: найдено только 3 прототипа со статусом "Давно решал"`
- `Подобрано 2 из 6: найдено только 2 прототипа со статусом "Нестабильно решает"`

### 11.4. Explicit `proto` Rejection Messaging

Если teacher явно выбрал `proto`, а этот `proto` не проходит активный фильтр, UI обязан объяснить это явно.

Обязательные требования:
- UI должен сообщать, что задача не была добавлена именно потому, что выбранный `proto` не проходит активный фильтр;
- UI не должен делать вид, что клик просто "не сработал";
- UI не должен молча подменять фильтровый запрос нейтральным подбором.

Примеры сообщений:
- `Этот прототип не попадает в фильтр "Не решал / мало решал"`
- `Этот прототип не попадает в фильтр "Давно решал"`
- `Этот прототип не попадает в фильтр "Нестабильно решает"`

### 11.5. `Reset / Rebuild` Explanation

UI должен объяснять teacher, что после явной пересборки набор может измениться.

Обязательные требования:
- teacher должен понимать, что без `reset / rebuild` выдача стабильна;
- teacher должен понимать, что после `reset / rebuild / смены ученика` может прийти другой набор по тем же критериям;
- это изменение должно подаваться как нормальное поведение, а не как нестабильность системы.

Пример текста:
- `После сброса или пересборки могут появиться другие задачи по тем же критериям`

### 11.6. No Silent Fallbacks

UI не должен скрывать продуктовую семантику teacher-фильтра.

Обязательные требования:
- UI не должен молча снимать фильтр, если кандидатов мало или нет;
- UI не должен молча подмешивать задачи из других фильтровых состояний;
- UI не должен создавать впечатление, что система "сама передумала".

Если задач по фильтру нет:
- UI должен показывать нулевой результат как честный outcome по выбранному фильтру, а не как пустую нерасшифрованную ошибку.

Пример текста:
- `По фильтру "Давно решал" в выбранной теме задачи не найдены`

### 11.7. Global `Выбрать все`

При использовании глобальной кнопки `Выбрать все` UI должен сохранять ту же прозрачность, что и для обычного подбора.

Обязательные требования:
- teacher должен понимать, что глобальная кнопка добавляет не "все задачи", а по одной задаче из каждой темы, где это возможно;
- если по части тем candidates отсутствуют, UI может показать суммарный shortage, но не должен создавать впечатление, что темы были просто пропущены без причины.

Пример текста:
- `Добавлено 7 задач: по одной из 7 тем, где фильтр нашёл кандидатов`

## 12. Non-Goals

Документ пока не покрывает:
- smart homework mixed-mode;
- одновременное комбинирование нескольких teacher-фильтров;
- будущую логику `Работы на занятии`;
- SQL / RPC implementation details.

## 13. Open Questions

Сюда выносятся остаточные продуктовые вопросы, которые ещё не утверждены.

На момент фиксации этой версии спецификации блокирующих open questions для manual teacher-picking не осталось.

Это означает:
- базовая система teacher-фильтров утверждена;
- педагогические цели трёх фильтров утверждены;
- правила `proto / topic / section` утверждены;
- правила `global Выбрать все`, shortage и `reset / rebuild` утверждены;
- UI-прозрачность зафиксирована на уровне продуктовой нормы.

Все вопросы, не вошедшие в текущую спецификацию, считаются сознательно вынесенными за рамки `v1` и относятся к разделу `Non-Goals`.

К таким вопросам относятся:
- smart homework mixed-mode;
- одновременное комбинирование нескольких teacher-фильтров;
- будущая логика `Работы на занятии`;
- SQL / RPC implementation details;
- любые расширения, требующие изменения текущих педагогических целей фильтров.

Если в будущем появится новая продуктовая развилка, она должна оформляться не как неявное расширение этой версии, а как отдельное изменение спецификации:
- либо через обновление `teacher_picking_filters_v1_spec.md`;
- либо через новую версию контракта (`v2`), если смысл фильтров изменится.
