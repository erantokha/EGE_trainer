**Правило «квадратное vs вынесение»** (фиксировано оператором): полное квадратное (3 члена,
дискриминант) → *Сведе́ние к квадратному*; нет свободного члена (`a t²+b t=0`) → *Вынесение*; нет
среднего члена (`a t²+c=0`) → *Сведе́ние к квадратному* (по умолчанию). Полнота **консистентна внутри
источника** — ни один источник не расщепляется между полками.

> Нумерация: **источник 13.NN** (исходная задача) ≠ **Б13-0NN** (клон). Напр. источник 13.16 =
> «косинусы гаснут → простейшее», а Б13-016 = квадратный клон из семейства qfull. Не путать.

## Ключевые архитектурные решения
1. **Класс и Метод — реальные измерения каталога (бэкенд), НЕ фронт-группировка.** Причина: оператор
   хочет статистику **по методам** (как подтемы части 1 собираются в бэке). Фронт-группировки
   недостаточно — свёртка по методу не соберётся. → каталог-схема/RPC растут (red-zone, волна W13.1).
2. **Приёмы — перекрёстный ФАСЕТ** (мульти-метки: «формулы приведения», «осн. тождество» и т.п.; у
   задачи их несколько, один приём встречается в разных методах). **Данные несём сразу** (в банке уже
   размечено, `_classification_spec.md`), а **аналитику-по-приёмам — отдельной будущей волной** (это
   не уровень дерева, а фасетная свёртка). Не загонять себя в угол: теги в данных с дня 1.
3. **Баллы — единая первичная шкала.** №13 добавляет свои 0/1/2 в **тот же** градусник; таблицу
   первичный→вторичный расширяем с **70 → 100** (по мере добавления части 2). Сейчас scoring
   захардкожен «12×1» (`picker_stats.js`, `SECONDARY_BY_PRIMARY`) — обобщаем на per-task макс-балл.
4. **Двухуровневая проверка — одна модель попытки:** поля `self_score` (ученик) + `teacher_score`
   (учитель) + `status` (ждёт проверки / подтверждено). Два прогноза в градуснике: **«самооценка»**
   (предварительный, для самоподготовки) и **«подтверждённый»** (официальный, учительский). Путь
   подтверждения — через **ревью ДЗ** (`renderReviewCards`, `hw.js`) — там единственный готовый
   per-question учительский экран. Свободные попытки части 2 = только самооценка (предварительно);
   официальными становятся через ДЗ. НЕ плодить отдельные сущности.
5. **Окружность отбора — SVG файлом через `<img src=…svg>`** (безопасно, как фигуры части 1; печатается;
   без риска `safe_dom`). Перекраска стиля = перегенерация `circle_svg` → пересборка SVG-файлов. **НЕ**
   inline-SVG (зависимость от `safe_dom`, security-review).
6. **Эталон-решение** в задаче = пошаговая **LaTeX-цепочка** (рендерится MathJax в DOM, как stem
   части 1) + **SVG-окружность** (`<img>`) + ответ. Визуальный стиль окружности уже утверждён и запечён
   в `circle_svg` (токены оператора).

## Зафиксированные решения оператора (2026-06-17)
- **Шкала — единый градусник, официальная шкала ФИПИ 0-32 первичных → 0-100 вторичных.** Текущая таблица
  `SECONDARY_BY_PRIMARY` (`picker_stats.js:121-135`) содержит официальные строки 0-12 (→70). Расширяется
  официальными строками 13-32 по мере добавления части 2 (потолок растёт к 100). НЕ интерполяция, не своя
  кривая — официальные значения из того же источника (монотонность гарантирована). **Полная таблица
  (получена от оператора 2026-06-17; строки 0-12 идентичны существующим в `picker_stats.js` — подтверждает
  официальность):**
  ```
  перв→втор: 0→0 1→6 2→11 3→17 4→22 5→27 6→34 7→40 8→46 9→52 10→58 11→64 12→70
             13→72 14→74 15→76 16→78 17→80 18→82 19→84 20→86 21→88 22→90 23→92 24→94
             25→95 26→96 27→97 28→98 29→99 30→100 31→100 32→100
  ```
  Плато на вершине: **30/31/32 → 100** (вторичный упирается в 100 уже с 30 первичных — учесть в
  отображении прогноза/цели). Механика W13.2: снять кламп `Math.min(12,…)` в
  `secondaryFromPrimary`/`secondaryFromPrimaryExact`/термометре; `updateScoreForecast` считать по per-task
  `max_primary` (№13=2) вместо «+1 за задание». До добавления всей части 2 достижимый потолок < 100 —
  ожидаемо.
- **Модерация учителей — НЕ делаем** (осознанное продуктовое решение: ожидаются тысячи учителей, ручная
  апробация неоправданна). Граница безопасности = согласие ученика (consent accept) + отзыв доступа.
  Самоэскалация роли (`update_my_profile`→`teachers.approved=true`) остаётся как есть.
- **Consent-UI hardening — отложен на «перед большим продакшеном», НЕ сейчас.** Признан нужным, но не
  блокирует часть 2. Пакет (отдельная гигиеническая волна когда-нибудь): (1) анти-impersonation в экране
  подтверждения — показывать реальный email учителя + предупреждение + запрет имён-самозванцев
  («Поддержка»/«Администрация»); (2) видимый список «Мои учителя» + отзыв на видном месте + уведомление
  ученику о новой привязке; (3) rate-limit исходящих инвайтов учителя.
- **W13.2 (внутри scope волны, обязательно):** новый teacher-write балла гейтить на accepted-связь
  (`is_teacher_for_student`/`teacher_students`), скоуп «только свои назначенные ДЗ этому ученику»,
  аудит-след (кто/когда поставил `teacher_score`), видимость балла и автора ученику. Так «ошибочный accept»
  = видимое/атрибутируемое/исправимое, а не тихий вандализм статистики. **Предусловие W13.2:** выгрузить
  реальные прод-дефиниции `is_teacher_for_student`/`is_allowed_teacher`/RLS `profiles`/`teachers` в
  `docs/supabase/` (сейчас governance-дрейф: вызываются в 20+ местах, в репо не определены).
