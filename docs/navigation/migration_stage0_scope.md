# Этап 0. Рамка этапа и Definition of Done

Дата обновления: 2026-03-29

Этот документ фиксирует рамку этапа 0 для окончательного перехода на 4-слойную архитектуру. Цель этапа 0: не переписать продукт, а зафиксировать канонический контракт, инвентарь runtime-RPC и правила владения, чтобы дальнейшая миграция шла по понятному и проверяемому плану.

## Цель этапа

На выходе этапа команда должна иметь один согласованный ответ на четыре вопроса:
- что считается целевой 4-слойной архитектурой;
- какие runtime-RPC реально используются продуктом;
- где лежит канонический SQL-источник каждой runtime-функции;
- кто отвечает за каждую runtime-функцию.

## Что входит в этап 0

- Короткий архитектурный контракт слоёв 1-4 без переписывания всей существующей документации.
- Формальные определения доменных сущностей `theme`, `subtopic`, `unic`, `question`.
- Формальные правила для продуктовых состояний и метрик, как минимум: `coverage`, `solved`, `weak`, `stale`.
- Полный реестр runtime-RPC, от которых зависит фронт и `app/providers/*`.
- Явное разделение runtime-RPC на канонические имена и временные legacy-алиасы.
- Выгрузка и коммит канонического SQL для каждого runtime-RPC в репозиторий.
- Фиксация owner для каждого runtime-RPC.
- Чек-лист приёмки этапа 0.

## Что не входит в этап 0

- Перевод экранов на новые read-контракты.
- Переписывание frontend-логики статистики, рекомендаций или picking.
- Достройка слоёв 2-4 в рантайме.
- Удаление legacy-RPC, legacy-fallback и legacy-таблиц.
- Перенос write-path с `attempts` на новый канонический контур.
- Массовая переработка существующего SQL, если это не требуется для фиксации канонического источника.
- Backfill, производственные миграции данных и оптимизация производительности, кроме минимальной проверки актуальности SQL.

## Обязательные артефакты этапа

- Документ с архитектурным контрактом слоёв 1-4: [architecture_contract_4layer.md](architecture_contract_4layer.md).
- Реестр runtime-RPC с полями `canonical_name`, `aliases`, `used_by`, `source_sql_file`, `owner`, `status`.
- Канонический SQL-файл в репозитории для каждого runtime-RPC.
- Явная фиксация owner для каждого runtime-RPC.
- Чек-лист приёмки этапа 0.

## Правила фиксации

### 1. Правило runtime-RPC

В реестр попадает любая RPC-функция, если от неё зависит runtime-поведение:
- экранов в `tasks/*`;
- провайдеров в `app/providers/*`;
- серверных read/write-контрактов, которые вызываются из клиентского кода напрямую или через thin-wrapper.

Не включаются служебные SQL-функции, если они не вызываются фронтом напрямую и не являются публичным runtime-контрактом.

### 2. Правило канонического имени

У каждой runtime-RPC должно быть одно каноническое имя. Если фронт использует fallback или исторические имена, они фиксируются как `aliases`, но не заменяют каноническое имя.

### 3. Правило SQL-источника

Для каждой runtime-RPC в репозитории должен существовать отдельный SQL-файл с полным определением функции. Snapshot схемы, excerpt в markdown или знание "она есть в live DB" не считаются достаточным источником.

### 4. Правило owner

У каждой runtime-RPC должен быть ровно один основной owner. Допускается дополнительно указать reviewer или backup-owner, но отсутствие основного owner блокирует закрытие этапа.

### 5. Правило допустимых доказательств

Этап 0 считается закрытым только по данным, которые можно проверить в репозитории. Live Supabase может быть источником выгрузки, но не заменяет артефакты в git.

## Definition of Done

Этап 0 считается завершённым, только если одновременно выполнены все условия:

- Есть короткий архитектурный контракт слоёв 1-4, согласованный с текущей миграционной целью.
- Есть полный реестр runtime-RPC, от которых зависит фронт.
- У каждого runtime-RPC есть канонический SQL-файл в репозитории.
- У каждого runtime-RPC есть owner.

Дополнительные критерии приёмки:

- В реестре нет runtime-RPC со статусом `unknown`.
- Для каждого legacy-алиаса указан канонический runtime-RPC, который он прикрывает.
- Нельзя закрыть этап, если хотя бы одна функция используется фронтом, но существует только в live DB или только в schema snapshot.
- Нельзя закрыть этап, если owner определён "на команду" без конкретного ответственного.

## Чек-лист приёмки

- [ ] Утверждена рамка этапа 0: что входит, что не входит, что считается результатом.
- [ ] Утверждён список обязательных артефактов этапа 0.
- [ ] Утверждён формат реестра runtime-RPC.
- [ ] Утверждено правило "одна runtime-RPC -> одно каноническое имя -> один SQL-файл -> один owner".
- [ ] Подтверждено, что schema snapshot не считается достаточным SQL-источником.
- [ ] Подтверждено, что live Supabase без выгрузки в git не считается выполнением этапа.

## Блокеры этапа

Ниже перечислены блокеры, которые нужно признать заранее:

- Нет доступа к live Supabase, откуда можно выгрузить актуальные функции.
- Нет согласованного owner-моделя для runtime-RPC.
- Нет единого решения, какие legacy-алиасы допустимо сохранять временно.
- Нет согласованного места в репозитории для канонических SQL-файлов и runtime-реестра.

## Следующий шаг после утверждения этого документа

После утверждения рамки этапа команда переходит к следующему подэтапу:
- оформить короткий архитектурный контракт слоёв 1-4;
- собрать полный runtime-RPC inventory;
- выполнить gap-analysis по SQL-покрытию;
- назначить owner и зафиксировать это в репозитории.
