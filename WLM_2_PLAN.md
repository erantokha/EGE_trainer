## §1. Цель

В уже существующем режиме занятия на `tasks/list.html` дать учителю **на каждой карточке** проставить
один из четырёх флагов разбора и (опционально) тег навыка из единого словаря. Это **учительская
приватная оценка** — не видна ученику. Данные ложатся в новую таблицу событий карточки и копят
кейстоун-сигнал для будущего умного ДЗ, отчёта и диагностики по навыкам. Без потребителей этих данных
в этой волне — только корректный сбор.

## §2. Контекст и мотивация

- WLM.1 построил поверхность занятия (тумблер на `list.html`) и контейнер занятия/конспекта. Флаги
  были сознательно вынесены из WLM.1 — это их волна.
- Флаги — **кейстоун** всей аналитики (`PRODUCT_VISION.md`, «Приоритеты» п.1): на них опираются умное
  ДЗ, качественная часть отчёта, диагностика по навыкам. Их нужно начать собирать заранее, чтобы к
  моменту появления потребителей была история.
- Бинарный «верно/неверно» теряет педагогически ключевое — *как* ученик справился. Флаг оцифровывает
  суждение учителя (лучший датчик в комнате), сейчас выбрасываемое.
- Теги навыков слой 1 (ручные, дёшево) — первый слой будущей диагностики «атом, а не номер», без
  контентного банка атомов. Требуют **единого управляемого словаря** с самого начала, иначе данные
  не агрегируются.

## §3. Out of scope

- **Потребители флагов:** умное ДЗ, отчёт, диагностика — другие волны. Здесь только сбор.
- **Показ флагов ученику** — флаги приватны для учителя; ученик их не видит вообще.
- **Авто-таймер как фича/UI и любая модель длительности.** Захватываем только **пассивные
  timestamps** (`opened_at`/`flagged_at`) best-effort, без UI и без потребителя; если «открытие
  карточки» на списке не определяется чисто — оставляем `time_ms` null, не блокер.
- **Редактирование словаря навыков через UI.** Оператор сидит/расширяет словарь через SQL; админки нет.
- **Локированный измерительный режим**, режим B (самостоятельная работа), вторая часть, консолидация кабинета.
- Любые изменения потока конспекта/снимков из WLM.1, picker-движка, auth-flow, общего каркаса.

**§5.3. Провайдер.** Расширить провайдер занятия (§5.1): обёртки upsert/load событий + чтение словаря

(занятие длинное, перезагрузка вкладки не должна терять флаги). Пассивные timestamps: `opened_at` —
первое взаимодействие с карточкой в режиме занятия (фокус/клик), `flagged_at` — при простановке флага;
`time_ms` best-effort, при невозможности чисто определить «открытие» — null.

**§5.7. Bump + governance + отчёт.** `node tools/bump_build.mjs`. Прогон governance (§9). Скриншоты
(§9). `reports/wlm_2_report.md` с фактами и инструкцией оператору по применению бэкенда.

## §6. Данные / контракты / миграции

**Backup:** не нужен (только новые объекты). **Runtime-контракты:** новые RPC → регистрация в реестре +
`check_runtime_rpc_registry.mjs`.

**Схема (design-предложение; привязку к контейнеру занятия исполнитель уточняет по §5.1):**

```sql
-- словарь навыков (dimension, seed расширяет оператор)
create table if not exists public.skill_tags_dim (
  code text primary key,
  label text not null,
  topic text,                     -- группировка (триг / планиметрия / алгебра ...)
  sort int not null default 0,
  is_enabled boolean not null default true
);

-- событие карточки на занятии (флаг + теги + soft-таймстемпы)
create table if not exists public.lesson_items (
  id uuid primary key default gen_random_uuid(),
  konspekt_id uuid not null references public.konspekts(id) on delete cascade,  -- = контейнер занятия (уточнить имя в §5.1)
  question_id text not null,
  flag text check (flag in ('clean','hint','arith','lost')),  -- ✅ / 💡 / ⚠️ / ❌
  skill_tags text[] not null default '{}',
  opened_at timestamptz,
  flagged_at timestamptz,
  time_ms int,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  unique (konspekt_id, question_id)
);
alter table public.lesson_items enable row level security;
```

**RLS-инварианты (ключевые):**
- `lesson_items`: доступ **только** учителю-владельцу контейнера занятия (через ту же проверку
  ownership+consent, что и контейнер в WLM.1). **Ученик доступа НЕ имеет** — флаги приватны.
- `skill_tags_dim`: read для authenticated (это справочник), write — нет (seed/расширение через SQL оператором).

**RPC (layer-4, `security definer`, revoke anon / grant authenticated):**
- `lesson_item_upsert_v1(p_konspekt_id uuid, p_question_id text, p_flag text, p_skill_tags text[], p_opened_at timestamptz, p_flagged_at timestamptz) → lesson_items` — upsert по (контейнер, question_id), гейт по ownership+consent.
- `lesson_items_for_konspekt_v1(p_konspekt_id uuid) → setof lesson_items` — загрузка флагов занятия.
- `skill_tags_dim_v1() → setof skill_tags_dim` — словарь навыков (is_enabled = true).

**Seed стартового словаря (расширяет оператор; пометить в отчёте как «стартовый, требует ревью»):**
дроби · дискриминант · корни и радикалы · метод интервалов (анализ знаков) · формулы приведения ·
формулы двойного угла · тригонометрическая окружность / отбор корней · ОДЗ · преобразование выражений ·
базовая планиметрия (факты) · потеря/появление корня · знак при переносе.

**Sync screen-spec:** опционально `docs/navigation/lesson_flags_v1_spec.md` (кратко). Не блокер DoD.

## §7. Риски и stop-ask точки

**RED-ZONE:** новый RLS/ownership, новые runtime-RPC, привязка к существующему контейнеру занятия WLM.1.

- **Главный инвариант: флаги приватны для учителя.** Ученик не должен получить доступ к `lesson_items`
  ни через RLS, ни через RPC, ни через ученическую страницу конспектов. Любая попытка отдать флаги
  ученику — stop-ask.
- **Не сломать WLM.1.** Поток конспекта/снимков, тумблер, контейнер занятия — не менять; только
  добавлять. Если для привязки `lesson_items` пришлось бы менять контракт контейнера — stop-ask 10(а).
- **Scope lock.** Править только файлы §4. Запрещено: ученическая `konspekts.html`, `header.js`,
  picker-движок, auth-flow, общий `tasks/trainer.css` (только per-page), governance-скрипты, существующие RPC.
- **Stop-ask (сверх §6.3):** см. п.10 блока «Режим работы».
- **Регрессии вёрстки:** стили флагов — per-page; скриншоты обязательны (§9). **`?v=`:** обязателен `bump_build`.

## Режим работы: автономный

Не останавливайся за подтверждением на каждом шаге, не проси промежуточного ревью. Доведи до DoD (§8),
верни `reports/wlm_2_report.md` + completion summary. Куратор принимает работу целиком.

**Останавливайся (stop-ask) только в экстренных случаях:**
1. Попытка изменить файл вне §4.
2. Попытка зайти в §3 Out of scope или red-zone §7 без explicit approval.
3. План противоречит реальности кода (контейнер/RPC/сигнатура не та; реестр разошёлся).
4. DoD объективно недостижим без выхода за scope.
5. Governance-скрипт упал, причина не очевидна из diff волны.
6. Уязвимость/утечка креденшлов.
7. Задача распадается на 2+ независимых.
8. Один тест/сценарий упал 2+ раза подряд после починок, причина неясна.
9. Архитектурное решение влияет на модули вне §4.
10. **Проектная специфика WLM.2:** (а) фактическая модель WLM.1 из §5.1 не даёт контейнер-занятие для
    привязки `lesson_items` или использует иной идентификатор карточки — остановись с вариантами;
    (б) любая ситуация, где флаги становятся доступны ученику; (в) для привязки событий пришлось бы
    менять контракт/таблицы контейнера WLM.1.

**НЕ экстренное (работай сам):** мелкие развилки внутри scope; имена/селекторы; порядок шагов §5;
повторный прогон governance/smoke; желание показать промежуточный результат.

**Формат stop-ask:** короткое сообщение — какой пункт сработал, что обнаружено, варианты, рекомендация. Жди решения.

## §8. Критерии приёмки (DoD)

**Уровень A — готово исполнителем сейчас (без живого бэкенда):**
- A1. `docs/supabase/lesson_items.sql`: `lesson_items` + RLS (учитель-only, ученик без доступа) +
  `skill_tags_dim` + seed + 3 RPC; идемпотентно; RPC в реестре; `check_runtime_rpc_registry.mjs` зелёный.
- A2. §5.1-разведка зафиксирована в отчёте; привязка `lesson_items` к реальному контейнеру обоснована.
- A3. `tasks/list.js`/`list.html`: в режиме занятия на карточке 4 флаг-кнопки + дропдаун навыка; вне
  режима занятия не показываются; клиентская логика реализована против контрактов §6.
- A4. Провайдер расширен (upsert/load/словарь); словарь читается из БД, не хардкод.
- A5. Персистенция флагов при повторном входе реализована; пассивные timestamps пишутся.
- A6. Governance зелёные; `print-features` без регрессий; `bump_build` прогнан; скриншоты приложены.

**Уровень B — живая приёмка ПОСЛЕ применения бэкенда оператором:**
- B1. Учитель в режиме занятия флажит 2–3 карточки и ставит теги → перезагрузка вкладки → флаги/теги на месте.
- B2. Ученик НЕ видит флаги нигде (страница конспектов, прямые запросы) — RLS негатив.
- B3. Словарь навыков в дропдауне приходит из БД; добавление кода в `skill_tags_dim` оператором появляется в UI.
- B4. e2e teacher: проставление/персистенция флагов; e2e-негатив: ученик без доступа к `lesson_items`.
