## 1. Корень (заземлён куратором)
С выбранным учеником каждый клик «Выбрать всё»/«+» → `scheduleSyncAddedTasks` (debounce **90 мс**, `picker.js:3192`) → `syncAddedTasksToSelection` (`:3647`) → student-aware resolve (`loadTeacherPickingScreenV2 mode:'resolve'` `:3342` / `loadTeacherPickingResolveBatchV1` `:3424`), который внутри пересчитывает полное состояние ученика по всему каталогу. Проблемы:
- **B. По resolve на клик, без коалесинга:** 90 мс не склеивает ручные клики → N кликов → N тяжёлых resolve.
- **C. Нет in-flight-гарда:** есть только `_ADDED_SYNC_SEQ` (`:3655`) — он отбрасывает устаревший РЕЗУЛЬТАТ, но НЕ мешает слать перекрывающиеся запросы → конкурентные тяжёлые resolve → **500/510**.
- **Render-хвост:** предпросмотр дочитывает манифесты **последовательно** (`buildPreviewQuestionsFromResolveRows` `:4064`, `for…await getTeacherResolveManifestIndex` `:4143/4121`; функция кэширует, `:4027`).
- **D. «Начать»** (`#htRecStart`, enable/disable `:1054/1435`) завязана на завершение синка/предпросмотра, хотя `question_id` известны сразу после resolve.

Без ученика путь обрывается рано (`if (!sid) return null` `:3306`) → дёшево.

## 2. Цель
N кликов → **1–2** непересекающихся resolve; **ноль 500/510** от конкуренции; предпросмотр дочитывается **параллельно**; «Начать» активна, как только известны `question_id`. Поведение подбора (что выбирается) — **корректное и эквивалентное по смыслу** (тот же count/distinct/scope/фильтр; уже добавленные задачи сохраняются), см. §7.

## 5. Дизайн (что именно реализовать)

**5.1. Коалесинг + in-flight-гард (conflated loop).** Ввести контроллер вокруг существующего `syncAddedTasksToSelection`:
- state: `_syncInFlight=false`, `_syncDirty=false`, `_syncDebounceT`.
- `scheduleSyncAddedTasks(opts)`: всегда метить `_syncDirty=true`; `immediate` → сразу `maybeRunSync()`; иначе перезапустить debounce (**вынести период в константу, поднять до ~250–350 мс**) → по срабатыванию `maybeRunSync()`.
- `maybeRunSync()`: если `_syncInFlight` → выйти (dirty уже стоит). Иначе `runSync()`.
- `runSync()`: `_syncInFlight=true; _syncDirty=false;` → `await syncAddedTasksToSelection(...)` (существующая, с её `_ADDED_SYNC_SEQ` — оставить как belt-and-suspenders) → `finally { _syncInFlight=false; if (_syncDirty) maybeRunSync(); }` (trailing-прогон по последнему состоянию).
- Инвариант: в полёте максимум ОДИН синк; после оседания — ровно один финальный по актуальным счётчикам.
- `flushTeacherAddedTasksSelection` (`:3203`, immediate на смене ученика) — провести через тот же контроллер (не плодить параллельный синк; если в полёте — пометить dirty и дождаться/перезапустить).

**5.2. Параллель манифестов в предпросмотре.** В `buildPreviewQuestionsFromResolveRows` ПЕРЕД assembly-циклом: собрать все уникальные `manifest_path` из rows, **прогреть кэш параллельно** (`getTeacherResolveManifestIndex` через `Promise.all`/`mapLimit` с лимитом ~6) → затем существующий `for…await`-цикл попадает в кэш (без сетевого ожидания). Логику выбора задач НЕ менять — только префетч. (Паттерн `mapLimit` уже есть, `:761`.)

**5.3. Разблокировка «Начать».** Исследовать гейт `#htRecStart`. Цель — активировать кнопку, как только известны `question_id` (после resolve), не дожидаясь полной отрисовки предпросмотра. Если чисто отделяется — отделить (хранить resolved ids → enable; render предпросмотра идёт следом/параллельно). Если сильно связано — минимум: после §5.2 render быстрый, поэтому enable сразу по завершении (быстрого) синка; навигация «Начать» обязана получать валидные ids (не сломать). Если разделение требует выхода за picker.js или ломает навигацию → STOP-ASK.

**5.4. (опц.) AbortController** для устаревших fetch'ей манифестов/resolve — клиент перестаёт ждать вытесненный запрос. **Честно отметить в коде/отчёте: это клиентская отмена, серверный SQL не отменяет** (защита — в §5.1, не тут). Делать только если не усложняет.

## 7. Риски и stop-ask
- **Эквивалентность подбора (главное):** уже добавленные задачи **сохраняются** (incremental-дельта + исключения не трогаем). Но из-за коалесинга НОВЫЕ задачи дельты подбираются **меньшим числом бóльших батчей** → конкретные `question_id` новых задач МОГУТ отличаться от старого «по одной». Это **допустимо и ожидаемо**: count/distinct/scope/фильтр сохранены, resolve детерминирован по сиду. **Зафиксировать в отчёте.** Если charnet/тест пинит точные id выбранных задач — он сдвинется **by design**; проверить, что сдвиг только в идентичности (валидность/счётчики целы), и перебазировать golden.
- **Не сломать immediate/flush** (смена ученика, `onTeacherContextChanged immediate:true`): должен отрабатывать через контроллер, без гонок и без потери финального состояния.
- **«Начать»** не должна активироваться без валидных ids (нельзя пускать на пустую/недорешённую выборку).
- **Не уехать в SQL/уровень 2.** Любая правка вне picker.js/bump → STOP-ASK.

## 6.3 Режим работы: автономный
> **Автономный.** Доведи до DoD + замеры + отчёт. Деплой (FE push) — куратор. Stop-ask: 1) правка вне picker.js (кроме bump); 2) любой заход в SQL/RPC/уровень-2; 3) разблокировка «Начать» требует выхода за picker.js/ломает навигацию; 4) подбор стал НЕ эквивалентным (изменился count/scope/фильтр, а не только конкретные id); 5) charnet красный не из-за ожидаемого id-сдвига; 6) DoD недостижим; 7–9 стандартные. Не экстренное: точный период debounce (250–350); лимит mapLimit; формат замеров.

## 8. Критерии приёмки (DoD)
1. In-flight-гард: в полёте ≤1 синк; «Выбрать всё ×4 подряд» (с учеником) → **1–2** resolve-RPC (Network), не 4; **ноль 500/510**.
2. Debounce поднят и вынесен в константу; быстрые клики коалесятся.
3. Манифесты предпросмотра дочитываются **параллельно** (префетч), assembly не делает сетевых await по одному.
4. «Начать» активна по факту известных `question_id` (или сразу по завершении ускоренного синка); навигация не сломана.
5. Подбор эквивалентен по смыслу (count/distinct/scope/фильтр; добавленные сохраняются). Сдвиг конкретных id — задокументирован; charnet перебазирован при необходимости с обоснованием.
6. `flushTeacherAddedTasksSelection` (смена ученика) работает корректно через контроллер.
7. governance зелёный; `node --check picker.js`; bump_build; `git diff` — только picker.js (логика) + `?v=`.
8. **Замеры до/после** в отчёте (см. §9).

## 10. Зачем именно так
Пользователю важно только ФИНАЛЬНОЕ состояние выборки. Значит легально считать только его (debounce), один раз (in-flight), игнорируя устаревшее (seq). Мы не ускоряем сам запрос — мы перестаём слать заведомо лишние. Параллель манифестов убирает последовательный render-хвост. «Начать» по ids — потому что контент дорисуется на странице задач. Базовая стоимость ОДНОГО resolve остаётся (это уровень 2) — честно отразить в замерах.
