# W13.2 — Баллы части 2 + двухуровневая проверка (ученик → учитель)

**RED-ZONE, самая тяжёлая волна** (scoring, write-path, RLS/RPC, агрегаты). Контракт и решения —
`docs/navigation/part2_integration_contract.md`; разведка — `reports/part2_recon/RECON.md`. Предусловие:
закрыты **W13.0** (данные) и **W13.1** (показ/каталог).

## §1. Цель
Часть 2 начинает **считаться в градуснике** по модели «самооценка → подтверждение учителем»: ученик
ставит предварительный балл (0/1/2), учитель подтверждает/корректирует, в официальную статистику идёт
учительский балл; первичная шкала расширяется до **100**. Реализовать модель попытки `self_score /
teacher_score / status`, **два прогноза** («самооценка» / «подтверждённый»), учительский контрол в ревью ДЗ.

## §2. Контекст
`answer_events` хранит только `correct(boolean)` — ни балла, ни max. Scoring захардкожен «12×1»
(`picker_stats.js`: `SECONDARY_BY_PRIMARY`, цикл 1..12, `updateScoreForecast`); per-task max-score нет.
Запись: `write_answer_events_v1` (`source='test'` хардкод), ДЗ через `submit_homework_attempt_v2`→триггер.
У учителя по задаче — только SELECT-RLS, INSERT/UPDATE-политик нет; единственный per-question учительский
экран — `renderReviewCards()` (`hw.js`). Всё про двухуровневость — **greenfield** (0 строк в коде).

## §3. Out of scope
- НЕ менять структуру каталога/аккордеон (W13.1).
- НЕ строить аналитику по приёмам (отдельная будущая волна) — но `uses`-теги не ломать.
- НЕ автопроверять часть 2 (она ручная по определению).
- НЕ трогать scoring/правила части 1 кроме обобщения «per-task max» (часть 1 = свои значения, без регресса).

## §4. Затрагиваемые файлы (уточняется recon-шагом §5.2)
- БД/SQL: модель попытки части 2 (`self_score/teacher_score/status` — колонки или таблица), teacher-write
  RPC (security-definer + гейт `is_teacher_for_student`), RLS, агрегаты `student_analytics_screen_v1`.
  Источники — `docs/supabase/*`, `runtime_rpc_registry`, `stage9_*`.
- `tasks/picker_stats.js` (per-task max, `SECONDARY_BY_PRIMARY` до 100, два прогноза в градуснике).
- `tasks/trainer.js` (UI самооценки ученика 0/1/2 после показа эталона).
- `tasks/hw.js` (`renderReviewCards` — контрол «учитель подтверждает 0/1/2»).
- `app/providers/supabase-write.js` / RPC-слой (запись self/teacher score).

## §6. Данные / контракты / миграции
Все SQL/RLS/RPC — **red-zone**: через `docs/supabase/*` + `runtime_rpc_registry`, с approval оператора и
негатив-тестами RLS. Destructive SQL запрещён без explicit approval. Дрейф «репо vs прод» — проверять.

## §7. Риски и stop-ask
- **RED-ZONE по нескольким осям** (scoring, write-path, RLS, teacher-доступ): максимально узкий scope,
  усиленное evidence, **stop-ask перед каждой миграцией/RLS/RPC**.
- Безопасность: учитель не должен писать балл чужому ученику (гейт + RLS негатив-тест обязателен).
- Риск исказить градуснику часть 1 — §5.5/§5.9 регрессия обязательна.
- «Самооценка» не должна попадать в официальную статистику вместо подтверждённой — раздельные прогнозы.

## §8. Autonomy policy (`CURATOR.md §6.3`)
Stop-ask-confirm обязателен при: любой SQL/RLS/RPC-миграции; изменении write-path/scoring; teacher-доступе;
файлах вне §4; заходе в §3 Out of scope. Свободно: порядок §5, имена внутренних функций, верстка контролов
самооценки/подтверждения в пределах per-page. Никаких destructive операций и изменений части-1-логики без
явного одобрения.
