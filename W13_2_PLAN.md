# W13.2 — Баллы части 2 + двухуровневая проверка (ученик → учитель)

**RED-ZONE, самая тяжёлая волна трека** — затрагивает сразу несколько осей: scoring, Layer-1/write-path,
RLS, teacher-write RPC, агрегаты. Контракт и решения (**особенно раздел «Зафиксированные решения
оператора»**) — `docs/navigation/part2_integration_contract.md`; разведка — `reports/part2_recon/RECON.md`;
W13.1 — `reports/w13_1/W13_1_REPORT.md`.

**Предусловия:**
- W13.0 (данные) и W13.1 (каталог + показ + эталон) закрыты; **W13.1-SQL залит и пройден live-smoke** (№13
  виден и решается вживую). Без живого №13 двухуровневую проверку не приёмить.
- Выполнен **§5.0 governance-предусловие** (выгрузка auth-хелперов/RLS в репо) — см. ниже.

## §1. Цель
Часть 2 начинает **считаться в градуснике** по модели **«самооценка → подтверждение учителем»**: ученик
ставит предварительный балл (0/1/2), учитель подтверждает/корректирует, в **официальную** статистику идёт
**учительский** балл. Первичная шкала расширяется до **100** по официальной таблице ФИПИ. Реализовать:
модель попытки `self_score / teacher_score / status` (три состояния: сдано → оценено учеником → подтверждено
учителем), **два прогноза** в градуснике («самооценка» / «подтверждённый»), учительский контрол в ревью ДЗ —
с **обязательной обвязкой безопасности** (см. §5.4).

## §2. Контекст
- `answer_events` хранит только `correct(boolean)` — ни балла, ни max. Запись: свободная попытка →
  `write_answer_events_v1` (`source='test'` хардкод); ДЗ → `submit_homework_attempt_v2` → триггер
  `trg_homework_attempts_to_answer_events` (пишет `correct`-boolean).
- Scoring захардкожен «12×1»: `picker_stats.js` — `SECONDARY_BY_PRIMARY` (строки 0-12, клампы `Math.min(12,…)`
  в `secondaryFromPrimary`/`secondaryFromPrimaryExact`/`thermoColorByPrimary`), `updateScoreForecast` суммирует
  по +1 за задание. Per-task max-score нет.
- Агрегаты: `student_topic_state_v1`/`student_proto_state_v1` и teacher-роллапы считают `correct`. Teacher-роллапы
  (`teacher_topic_rollup_v1`/`type_rollup_v1`) читают **`public.question_bank`** (не каталог) — №13 туда уже
  залит в W13.1, но по `correct`-семантике.
- У учителя по данным ученика — **только SELECT-RLS** (`answer_events_select_teacher_students`), INSERT/UPDATE-политик
  нет; запись учителем в данные ученика возможна только через security-definer RPC с гейтом. Единственный
  per-question учительский экран — `renderReviewCards()` (`hw.js`).
- **Governance-дрейф (критично для этой волны):** гейт-хелперы `is_teacher_for_student`/`is_allowed_teacher`/
  `is_teacher` и RLS-политики `profiles`/`teachers`/`answer_events` **не выгружены в репо** (вызываются в 20+
  местах, определений нет) — без них новый teacher-write-гейт нельзя отревьюить из кода. См. §5.0.
- Всё про двухуровневость — **greenfield** (0 строк в коде).

## §3. Out of scope
- НЕ менять структуру каталога/аккордеон/показ (W13.0/W13.1).
- НЕ строить аналитику по приёмам (`uses`) — отдельная будущая волна; теги не ломать.
- НЕ автопроверять часть 2 (она ручная по определению).
- НЕ регрессировать scoring/правила части 1 (часть 1 = `correct`-boolean, свои значения).
- **Модерация учителей — НЕ делаем** (осознанное решение оператора; граница = consent ученика + отзыв).
- **Consent-UI hardening — НЕ здесь** (отложен на «перед большим продакшеном», контракт §«Зафикс. решения»).
- **Печать части 2** и **фото-прикрепление решения** (PRODUCT_VISION §4) — будущие отдельные поверхности.

## §5. Пошаговый план
> **Task-tracking (обязательно, `CURATOR.md §6.1`):** TaskList по §5.0–§5.11, обновлять статусы.

- **§5.0 Governance-предусловие (ДО любого teacher-write).** Выгрузить из прода реальные дефиниции
  `is_teacher_for_student`/`is_allowed_teacher`/`is_teacher` + RLS `profiles`/`teachers`/`answer_events` в
  `docs/supabase/`. Цель — новый write-гейт строится поверх читаемого, отревьюенного кода, а не «вслепую».
  **stop-ask, если выгруженные дефиниции расходятся с ожиданием** (репо↔прод-дрейф).
- **§5.1** Прочитать контракт (§«Зафикс. решения»)/recon/`stage9_*` (write-path)/`student_analytics_screen_v1_spec`.
- **§5.2 RECON write-path/scoring (read-only, внутри волны).** Точно замапить: что/куда пишется при свободном
  ответе и при ДЗ, кто owner, как агрегаты считают итог, где `question_bank` кормит teacher-роллапы.
  Спроектировать **минимальное** хранилище `self_score/teacher_score/status`. **Оценить дробление волны**
  (напр. W13.2a = шкала+самооценка ученика; W13.2b = teacher-write+RLS+агрегаты) — каждая red-zone-миграция
  узкой. **Вывод — на approval оператора; stop-ask перед любой SQL/RLS/RPC.**
- **§5.3 Модель попытки части 2.** `self_score` (ученик), `teacher_score` (учитель), `status`
  (`сдано`/`оценено учеником`/`подтверждено учителем`). Правило: **в официальную статистику идёт
  `teacher_score`** (`coalesce`), в прогноз «самооценка» — `self_score`; до подтверждения учителем балл
  части 2 = черновик. Миграция — после approval, идемпотентно, без destructive.
- **§5.4 Teacher-write + обвязка безопасности (контракт §«Зафикс. решения»).** Новый RPC (security-definer):
  гейт `is_teacher_for_student` (**accepted-связь** в `teacher_students`) **И** скоуп «только **своя
  назначенная** ДЗ-попытка этого ученика» **И** аудит-след (`teacher_id` + timestamp кто/когда поставил)
  **И** видимость балла и автора ученику. Ученик пишет только `self_score` (свой `auth.uid()`), учитель —
  `teacher_score`. **RLS-негатив-тесты обязательны** (см. §5.11).
- **§5.5 Scoring — официальная шкала (зафиксирована).** Дописать в `SECONDARY_BY_PRIMARY` официальные строки
  13-32 (полная таблица — контракт §«Зафикс. решения»; плато **30/31/32 → 100**); снять клампы `Math.min(12,…)`;
  `updateScoreForecast` — суммировать по **per-task `max_primary`** (№13 = 2) вместо +1. Часть 1 без регресса.
  NB: пока в банке не вся часть 2 — достижимый потолок < 100, это ожидаемо.
- **§5.6 Read-side агрегаты.** Для части 2 учитывать **балл** (`teacher_score`), а не `correct`-boolean:
  `student_topic_state_v1`/`student_analytics_screen_v1` + teacher-роллапы (путь `question_bank`) —
  `coalesce(teacher_score, …)` для части 2; часть 1 = `correct` без изменений.
- **§5.7 Два прогноза в градуснике.** «Самооценка» (`self_score`) и «Подтверждённый» (`teacher_score`,
  официальный) — раздельно; самооценка **не** подменяет официальную.
- **§5.8 UI ученика.** После эталона (W13.1) — выбор 0/1/2 (самооценка) → запись `self_score`. Свободные
  попытки = только самооценка (предварительно).
- **§5.9 UI учителя.** В `renderReviewCards` (`hw.js`) — контрол 0/1/2 «подтвердить/исправить» → teacher-write
  RPC → `teacher_score`; рядом показать **эталон** для сверки; ученику видно кто/когда подтвердил.
- **§5.10 Путь подтверждения = через ДЗ.** Зафиксировать поведение: свободные попытки части 2 = только
  самооценка; официальными становятся через проверку учителем в ДЗ. (Проверка свободных попыток — будущая
  поверхность, вне волны.)
- **§5.11 Регрессия + тесты.** Часть 1 scoring/запись/статистика — идентичны (обязательно). **RLS-негатив:**
  учитель не своего ученика → отказ; учитель по чужой ДЗ → отказ; ученик пишет `teacher_score` → отказ.
  Idempotent submit. Evidence: скриншоты самооценки ученика, подтверждения учителем, обоих прогнозов; вывод
  governance; SQL-проверки; негатив-тесты.

## §7. Риски и stop-ask
- **RED-ZONE по нескольким осям** (scoring, write-path, RLS, teacher-доступ): максимально узкий scope, усиленное
  evidence, **stop-ask перед каждой миграцией/RLS/RPC**. Рассмотреть дробление на W13.2a/W13.2b (§5.2).
- **Безопасность.** Teacher-write — только на accepted-ученика и свою ДЗ; RLS-негатив обязателен. NB
  `security-audit-2026-06-10`: самоэскалация роли «учитель» жива (граница держится на consent+отзыв) — новый
  write-путь **не должен её усиливать**; гейт строго на `teacher_students` (accepted), скоуп на свою ДЗ, аудит.
- Не исказить градусник части 1 — регрессия §5.5/§5.11 обязательна.
- Самооценка не должна попадать в официальную статистику вместо подтверждённой — раздельные прогнозы, в
  official только `teacher_score`.
- Шкала — **официальная таблица из контракта**, не самодельная интерполяция; плато 30-32→100 учесть.
- **stop-ask:** точная схема (колонки vs отдельная таблица) — только после approval recon §5.2; если §5.0
  выявит расхождение репо↔прод в гейтах — остановиться и согласовать.

## §8. Autonomy policy (`CURATOR.md §6.3`)

файлах вне §4; заходе в §3 Out of scope; destructive-операциях; любых изменениях логики части 1. Свободно без
спроса: порядок шагов §5, имена внутренних функций, вёрстка контролов самооценки/подтверждения в пределах
per-page CSS. Деплой SQL в прод — **только оператор** (gated, SQL первым, FE вторым).
